## Dependency Tracking

BlaC optimizes re-renders through **dependency tracking** - components only re-render when the specific state they care about changes. BlaC offers two strategies: automatic proxy-based tracking and manual explicit dependencies.

## Two Tracking Modes

### Automatic Tracking (Default)

Without specifying dependencies, BlaC uses **JavaScript Proxies** to automatically track which properties you access:

```typescript
const [state] = useBloc(UserCubit);
return <div>{state.profile.name}</div>;
// ✅ Auto-tracks profile.name
// ✅ Only re-renders when profile.name changes
// ✅ Ignores changes to profile.email, settings, etc.
```

**Tradeoffs:** Convenient and automatic, but adds Proxy overhead. Adding the Proxy makes accessing the value about 5x slower than accessing it on the object directly, however we are still in the range of tens of million reads per second. Compared to the cost of unnecessary re-renders in react, the Proxy overhead is negligible.

### Manual Dependencies

Explicit dependencies give you precise control and better performance:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.profile.name],
});
// ✅ Explicitly tracks profile.name
// ✅ No proxy overhead - better performance
// ✅ Clear intent and easier to debug
```

**Tradeoffs:** Requires manual specification, but offers the best performance and clarity. Ideal for performance-critical components when the Automatic Tracking does not suffice.

<GuideArticleNote title="How Comparison Works">
  BlaC compares dependency values using **`Object.is()`**. If any value changed,
  the component re-renders. Otherwise, it receives the same frozen state
  snapshot.
</GuideArticleNote>

## Common Patterns

### Single Property

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.userName],
});
```

### Multiple Properties

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [
    cubit.state.profile.name,
    cubit.state.profile.email,
  ],
});
```

### Computed Values (Getters)

```typescript
class UserCubit extends Cubit<UserState> {
  get emailIsValid(): boolean {
    return validateEmail(this.state.email)
  }
}

const [, cubit] = useBloc(UserCubit);
return <>Email is ${cubit.emailIsValid ? 'Valid' : 'Invalid'}</>
// Only re-renders when emailIsValid's RESULT changes
```

**Why getters are powerful:** If `state.email` changes, the component will only re-renders if the email string goes from `valid -> invalid` or changes from `invalid -> valid`.

### Generator Dependencies (Advanced)

For components with many dependencies (5+), generators can short-circuit the dependency array by exiting early, this can be useful to exit early and avoid creating more arrays in memory:

With a generator the dependency detection will stop as soon as it encounters an answer that changed, similar to a `for` loop with a `break`:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: function* (cubit) {
    yield* cubit.state.answers.map((answer) => answer.correct);
  },
});
```

In comparison, here is the same code but without a generator, here the `map` will always go through all answers, no matter what:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => cubit.state.answers.map((answer) => answer.correct),
});
```

## Real-World Usage

### Dashboard Widgets

```typescript
function SalesWidget() {
  const [state] = useBloc(DashboardCubit);
  return <div>Sales: {state.sales} | Revenue: ${state.revenue}</div>;
}
// Only rerenders when state.sales or state.ervenue changes
```

### List

Every `TodoItemRow` would re-render every time `selected` changes, but with the `dependencies` option we can ensure that it only re-renders if this item is selected/deselected:

```typescript
function TodoItemRow({ item }: { item: TodoItemData }) {
  const [{ selected }, { remove, select }] = useBloc(TodoList, {
    dependencies: (bloc) => [bloc.state.selected === item],
  });

  return (
    <tr className={item === selected ? 'danger' : ''}>
      <td>{item.id}</td>
      <td>
        <a onClick={() => select(item)}>{item.label}</a>
      </td>
      <td>
        <a onClick={() => remove(item)}>{RemoveIcon}</a>
      </td>
    </tr>
  );
}
```

### Computed Metrics

```typescript
class AnalyticsCubit extends Cubit<AnalyticsState> {
  get conversionRate(): number {
    return this.state.visitors > 0
      ? (this.state.conversions / this.state.visitors) * 100
      : 0;
  }
}

function MetricsDisplay() {
  const [, cubit] = useBloc(AnalyticsCubit);
  return <p>Conversion: {cubit.conversionRate.toFixed(2)}%</p>;
}
```

## Configuration

Proxy tracking is enabled by default. To disable globally:

```typescript
import { Blac } from '@blac/core';

Blac.setConfig({ proxyDependencyTracking: false });
```

When disabled, components without explicit `dependencies` re-render on every state change. Manual dependencies always work regardless of this setting.

## Key Takeaways

- **Two Modes**: Automatic (Proxy-based) vs Manual (Explicit control)
- **Automatic Works**: Even without explicit dependencies, BlaC optimizes via Proxy tracking
- **Manual Is Faster**: Explicit dependencies have better performance (no Proxy overhead) but is more verbose
- **Value Comparison**: Uses `Object.is()` for reference equality, not deep equality
- **Track Primitives**: Best results with strings, numbers, booleans, or computed getters
- **Getters Are Powerful**: Track derived values that only trigger re-renders when results change

<GuideArticleNote title="Production Strategy">
**Prototyping:** Start with automatic tracking for speed and convenience.

**Refine:** Add explicit dependencies to performance-critical components where simple "value changed" detection is too generic.

</GuideArticleNote>
