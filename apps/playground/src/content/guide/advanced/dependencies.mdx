## Dependency Tracking

BlaC optimizes re-renders through **dependency tracking** - components only re-render when the specific state they care about changes. BlaC offers two strategies: automatic proxy-based tracking and manual explicit dependencies.

## Two Tracking Modes

### Automatic Tracking (Default)

Without specifying dependencies, BlaC uses **JavaScript Proxies** to automatically track which properties you access:

```typescript
const [state] = useBloc(UserCubit);
return <div>{state.profile.name}</div>;
// ✅ Auto-tracks profile.name
// ✅ Only re-renders when profile.name changes
// ✅ Ignores changes to profile.email, settings, etc.
```

**Tradeoffs:** Convenient and automatic, but adds Proxy overhead. Great for prototyping and simple components.

### Manual Dependencies

Explicit dependencies give you precise control and better performance:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.profile.name],
});
// ✅ Explicitly tracks profile.name
// ✅ No proxy overhead - better performance
// ✅ Clear intent and easier to debug
```

**Tradeoffs:** Requires manual specification, but offers the best performance and clarity. Ideal for production and performance-critical components.

<GuideArticleNote title="How Comparison Works">
  BlaC compares dependency values using **`Object.is()`**. If any value changed,
  the component re-renders. Otherwise, it receives the same frozen state
  snapshot.
</GuideArticleNote>

## Common Patterns

### Single Property

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.userName],
});
```

### Multiple Properties

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [
    cubit.state.profile.name,
    cubit.state.profile.email,
  ],
});
```

### Computed Values (Getters)

```typescript
class UserCubit extends Cubit<UserState> {
  get fullName(): string {
    return `${this.state.firstName} ${this.state.lastName}`;
  }
}

const [, cubit] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.fullName],
});
// Only re-renders when fullName's RESULT changes
```

**Why getters are powerful:** If `firstName` changes from `"John"` to `"Jane"` and back to `"John"`, only the first change triggers a re-render. The second change doesn't because `fullName` returns to the same value.

### Generator Dependencies (Advanced)

For components with many dependencies (5+), generators can short-circuit evaluation:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: function* (cubit) {
    if (yield cubit.state.name) return; // Stop if name changed
    yield cubit.state.email; // Only check if name didn't change
    yield cubit.state.phone;
  },
});
```

## Real-World Usage

### Dashboard Widgets

```typescript
function SalesWidget() {
  const [state] = useBloc(DashboardCubit, {
    dependencies: (cubit) => [cubit.state.sales, cubit.state.revenue],
  });
  return <div>Sales: {state.sales} | Revenue: ${state.revenue}</div>;
}
// Doesn't re-render when user counts or other metrics change
```

### Form Sections

```typescript
function EmailSection() {
  const [state, cubit] = useBloc(ProfileCubit, {
    dependencies: (cubit) => [cubit.state.email],
  });
  return <input value={state.email} onChange={(e) => cubit.updateEmail(e.target.value)} />;
}
// Doesn't re-render when firstName, lastName, etc. change
```

### Computed Metrics

```typescript
class AnalyticsCubit extends Cubit<AnalyticsState> {
  get conversionRate(): number {
    return this.state.visitors > 0
      ? (this.state.conversions / this.state.visitors) * 100
      : 0;
  }
}

function MetricsDisplay() {
  const [, cubit] = useBloc(AnalyticsCubit, {
    dependencies: (cubit) => [cubit.conversionRate],
  });
  return <p>Conversion: {cubit.conversionRate.toFixed(2)}%</p>;
}
```

## Common Mistakes

### ❌ Tracking Entire Objects

```typescript
// BAD: Re-renders on ANY profile change
dependencies: (cubit) => [cubit.state.profile];

// GOOD: Track specific fields
dependencies: (cubit) => [cubit.state.profile.name, cubit.state.profile.email];
```

### ❌ Disabling Proxies Without Dependencies

```typescript
// BAD: Disables automatic tracking but doesn't use manual dependencies
Blac.setConfig({ proxyDependencyTracking: false });
const [state] = useBloc(AppCubit); // Now re-renders on EVERY change!

// GOOD: Keep proxies enabled (default) OR use manual dependencies
const [state] = useBloc(AppCubit, {
  dependencies: (cubit) => [cubit.state.user.name],
});
```

### ❌ Conditional Dependency Arrays

```typescript
// BAD: Unstable dependency structure
dependencies: (cubit) => {
  const deps = [cubit.state.name];
  if (someCondition) deps.push(cubit.state.email); // Inconsistent!
  return deps;
};

// GOOD: Keep dependency arrays consistent
dependencies: (cubit) => [cubit.state.name, cubit.state.email];
```

## Best Practices

**Do:**

- ✅ Use automatic tracking for prototyping and simple components
- ✅ Use manual dependencies for production and performance-critical components
- ✅ Track primitive values (strings, numbers, booleans) or computed getters
- ✅ Keep dependency arrays simple and consistent
- ✅ Test with render counters to verify optimization

**Don't:**

- ❌ Track entire objects when you only need specific properties
- ❌ Disable proxy tracking globally without using manual dependencies
- ❌ Create conditional or unstable dependency arrays
- ❌ Perform expensive computations in the dependencies function itself

## Debugging Re-Renders

Verify your optimizations with a simple render counter:

```typescript
function MyComponent() {
  const renderCount = useRef(0);
  const [state] = useBloc(UserCubit, {
    dependencies: (cubit) => [cubit.state.userName],
  });

  useEffect(() => { renderCount.current++; });

  return (
    <div>
      <span>Renders: {renderCount.current}</span>
      <p>{state.userName}</p>
    </div>
  );
}
```

Change untracked properties and verify the counter doesn't increment!

## Configuration

Proxy tracking is enabled by default. To disable globally:

```typescript
import { Blac } from '@blac/core';

Blac.setConfig({ proxyDependencyTracking: false });
```

When disabled, components without explicit `dependencies` re-render on every state change. Manual dependencies always work regardless of this setting.

## Key Takeaways

- **Two Modes**: Automatic (Proxy-based) vs Manual (Explicit control)
- **Automatic Works**: Even without dependencies, BlaC optimizes via Proxy tracking
- **Manual Is Faster**: Explicit dependencies have better performance (no Proxy overhead)
- **Value Comparison**: Uses `Object.is()` for reference equality, not deep equality
- **Track Primitives**: Best results with strings, numbers, booleans, or computed getters
- **Getters Are Powerful**: Track derived values that only trigger re-renders when results change

<GuideArticleNote title="Production Strategy">
**Prototyping:** Start with automatic tracking for speed and convenience.

**Production:** Add explicit dependencies to performance-critical components. Proxy overhead adds up in large applications with hundreds of components.

**Best approach:** Mix both strategies - use automatic for simple components, manual for complex or high-frequency updates.

</GuideArticleNote>
