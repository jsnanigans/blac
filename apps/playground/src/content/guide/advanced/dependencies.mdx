## Introduction

Dependency tracking is the foundation of **fine-grained reactivity** in BlaC. By specifying exactly which parts of the state your component depends on, you can dramatically reduce unnecessary re-renders and improve application performance.

While BlaC tracks state changes automatically, dependency tracking allows you to be **explicit** about which state properties trigger re-renders. This is especially powerful for large state objects where most components only care about a small subset of the data.

### What You'll Learn

- How to use the `dependencies` option for fine-grained subscriptions

## Understanding Dependency Tracking

By default, when you use `useBloc` without dependencies, your component re-renders on **every** state change:

```typescript
// No dependencies - re-renders on ANY state change
const [state] = useBloc(UserCubit);
// ❌ Component re-renders when ANY part of state changes
```

With dependencies, you specify exactly which values to track:

```typescript
// Single dependency - re-renders only when name changes
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.profile.name],
});
// ✅ Only re-renders when profile.name changes
```

<GuideArticleNote title="How It Works">
BlaC compares the dependency array from the previous render with the current render using **shallow equality** (`Object.is`). If any value in the array has changed, the component re-renders.
</GuideArticleNote>

## Dependencies vs Selectors

BlaC provides two mechanisms for optimizing re-renders:

### Dependencies

Track specific state properties or computed values:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.userName],
});
```

**When to use:**
- You need explicit, easy-to-read subscription logic
- You're tracking primitive values (strings, numbers, booleans)
- You're tracking computed properties from getters
- You have large state objects with many properties

### Selectors

Derive new values from state with custom comparison:

```typescript
const [state] = useBloc(UserCubit, {
  selector: (state) => ({
    name: state.profile.name,
    email: state.profile.email,
  }),
});
```

**When to use:**
- You need to derive or transform state values
- You want to compare complex objects by value
- You need custom equality comparison logic
- The derived value is expensive to compute

**Both techniques complement each other.** Dependencies are simpler and more explicit, while selectors offer more flexibility for complex transformations.

## Basic Dependency Patterns

### Single Property Dependency

Track one specific property:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.profile.name],
});
// ✅ Only re-renders when name changes
```

### Multiple Dependencies

Track multiple properties:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [
    cubit.state.profile.name,
    cubit.state.profile.email,
  ],
});
// ✅ Re-renders when name OR email changes
```

### Computed Property Dependencies

Track derived values:

```typescript
class UserCubit extends Cubit<UserState> {
  // Computed property (getter)
  get fullName(): string {
    return `${this.state.firstName} ${this.state.lastName}`;
  }
}

// Track computed value
const [, cubit] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.fullName],
});
// ✅ Re-renders only when the RESULT of the computation changes
```

**Smart optimization:** If `firstName` changes from "John" to "Jane" but `lastName` stays "Doe", and then `lastName` changes from "Doe" to "Smith", the component only re-renders when the final `fullName` string actually changes!

## Advanced Patterns

### Deep Property Access

Track nested properties:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [
    cubit.state.settings.theme,
    cubit.state.settings.notifications,
  ],
});
// ✅ Only re-renders when theme or notifications change
// ✅ Ignores changes to settings.language or other properties
```

### Conditional Dependencies (Advanced)

Dynamically adjust dependencies based on conditions:

```typescript
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => {
    const deps = [cubit.state.profile.name];

    // Add email to dependencies only if notifications enabled
    if (cubit.state.settings.notifications) {
      deps.push(cubit.state.profile.email);
    }

    return deps;
  },
});
// ⚠️ Advanced: dependency array changes dynamically
```

<GuideArticleNote title="Warning">
Dynamic dependency arrays can be confusing and hard to debug. Prefer static dependency arrays when possible. If you need conditional logic, consider using multiple components or selectors instead.
</GuideArticleNote>

### Array and Object Dependencies

```typescript
// ❌ Common Mistake: Tracking entire objects
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.profile],
});
// Problem: Entire profile object in dependency array
// Re-renders on ANY profile property change (name, email, avatar)

// ✅ Better: Track specific properties
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [
    cubit.state.profile.name,
    cubit.state.profile.email,
  ],
});
// Solution: Explicitly list properties you care about

// ✅ Alternative: Use a selector for complex logic
const [state] = useBloc(UserCubit, {
  selector: (state) => ({
    name: state.profile.name,
    email: state.profile.email,
  }),
});
// Selector returns new object, compared by shallow equality
```

### Example 3: Computed Engagement Metrics

```typescript
class AnalyticsCubit extends Cubit<AnalyticsState> {
  get conversionRate(): number {
    const { visitors, conversions } = this.state;
    return visitors > 0 ? (conversions / visitors) * 100 : 0;
  }

  get averageSessionTime(): number {
    const { totalSessionTime, sessionCount } = this.state;
    return sessionCount > 0 ? totalSessionTime / sessionCount : 0;
  }
}

// Component only re-renders when computed metrics change
function MetricsDisplay() {
  const [, cubit] = useBloc(AnalyticsCubit, {
    dependencies: (cubit) => [
      cubit.conversionRate,
      cubit.averageSessionTime,
    ],
  });

  return (
    <div>
      <p>Conversion: {cubit.conversionRate.toFixed(2)}%</p>
      <p>Avg Session: {cubit.averageSessionTime.toFixed(0)}s</p>
    </div>
  );
}
```

## Performance Best Practices

<GuideArticleNote title="Do">
- ✅ Specify dependencies for fine-grained subscriptions
- ✅ Track primitive values and computed properties
- ✅ Use multiple dependencies when needed
- ✅ Combine with selectors for complex cases
- ✅ Test re-render behavior with render counters
- ✅ Keep dependency arrays static (not dynamic)
</GuideArticleNote>

<GuideArticleNote title="Don't">
- ❌ Track entire objects/arrays unless necessary
- ❌ Create overly dynamic dependency arrays
- ❌ Forget to specify dependencies (re-renders on all changes)
- ❌ Assume dependencies work like React's `useEffect` deps
- ❌ Mix dependencies and selectors in confusing ways
- ❌ Use dependencies for component-local state
</GuideArticleNote>

### Rule of Thumb

**Track primitives** (strings, numbers, booleans) and **computed values** in your dependency array. Avoid tracking entire objects or arrays unless you truly need to re-render on any change to that structure.

## Common Anti-Patterns

### ❌ Not Using Dependencies at All

```typescript
// BAD: No optimization
function UserDisplay() {
  const [state] = useBloc(AppCubit);
  // Re-renders on EVERY state change in the entire app!

  return <div>{state.user.name}</div>;
}

// GOOD: Specify dependencies
function UserDisplay() {
  const [state] = useBloc(AppCubit, {
    dependencies: (cubit) => [cubit.state.user.name],
  });

  return <div>{state.user.name}</div>;
}
```

### ❌ Tracking Too Much

```typescript
// BAD: Tracking entire state object
function Component() {
  const [state] = useBloc(AppCubit, {
    dependencies: (cubit) => [cubit.state],
  });
  // Defeats the purpose - re-renders on any state change!
}

// GOOD: Track specific fields
function Component() {
  const [state] = useBloc(AppCubit, {
    dependencies: (cubit) => [cubit.state.field1, cubit.state.field2],
  });
}
```

### ❌ Confusing Dependencies with React Deps

```typescript
// DON'T DO THIS - dependencies are NOT React dependency arrays!
const [state] = useBloc(UserCubit, {
  dependencies: (cubit) => [cubit.state.name], // ✅ Correct
});

// React useEffect dependencies are different:
useEffect(() => {
  // Effect logic
}, [state.name]); // Different purpose - controls effect execution
```

BlaC dependencies control **when to re-render**, while React dependencies control **when to run effects**.

## Debugging Re-Renders

Use a render counter to track re-renders:

```typescript
function RenderCounter({ label }: { label: string }) {
  const renderCount = useRef(0);
  useEffect(() => {
    renderCount.current++;
  });

  return (
    <span>
      {label}: {renderCount.current} renders
    </span>
  );
}

function MyComponent() {
  const [state] = useBloc(UserCubit, {
    dependencies: (cubit) => [cubit.state.userName],
  });

  return (
    <div>
      <RenderCounter label="Component" />
      <p>{state.userName}</p>
    </div>
  );
}
```

## Testing with Dependencies

```typescript
import { describe, it, expect } from 'vitest';
import { renderHook } from '@testing-library/react';
import { useBloc } from '@blac/react';

describe('Dependencies', () => {
  it('only re-renders on dependency changes', () => {
    const cubit = new UserCubit();
    let renderCount = 0;

    const { rerender } = renderHook(() => {
      renderCount++;
      return useBloc(UserCubit, {
        dependencies: (c) => [c.state.name],
      });
    });

    // Initial render
    expect(renderCount).toBe(1);

    // Change name - should re-render
    cubit.updateName('New Name');
    rerender();
    expect(renderCount).toBe(2);

    // Change email - should NOT re-render (not in dependencies)
    cubit.updateEmail('new@email.com');
    rerender();
    expect(renderCount).toBe(2); // Still 2!
  });
});
```

## Key Takeaways

- **Fine-Grained Subscriptions**: Dependencies enable precise control over re-renders
- **Shallow Equality**: BlaC uses `Object.is` to compare dependency values
- **Track Primitives**: Best results with strings, numbers, booleans, and computed values
- **Avoid Objects**: Don't track entire objects unless you need all properties
- **Static Arrays**: Prefer static dependency arrays over dynamic ones
- **Combine Techniques**: Use dependencies with selectors for maximum optimization
- **Test Behavior**: Use render counters to verify optimization
