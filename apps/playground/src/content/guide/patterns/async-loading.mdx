## The State Machine Pattern

The most robust way to handle loading states is with **discriminated union types**—a state machine where each state is explicit and type-safe.

### Why Discriminated Unions?

```typescript
// ❌ BAD: Boolean flags lead to impossible states
interface LoadingState {
  isLoading: boolean;
  hasError: boolean;
  data?: string;
  error?: string;
}
// Can be: loading=true AND hasError=true (impossible!)
// Can be: data=undefined AND hasError=false (what's the state?)

// ✅ GOOD: Discriminated union makes impossible states unrepresentable
type LoadingState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: string }
  | { status: 'error'; error: string };
// TypeScript narrows types based on status field
```

<GuideArticleNote title="Type Safety">
With discriminated unions, TypeScript **narrows the type** based on the `status` field. You can't access `data` unless `status === 'success'`, preventing runtime errors.
</GuideArticleNote>

### Implementation

```typescript
class DataFetchCubit extends Cubit<LoadingState> {
  constructor() {
    super({ status: 'idle' });
  }

  fetchData = async () => {
    // Guard: Prevent duplicate requests
    if (this.state.status === 'loading') return;

    this.emit({ status: 'loading' });

    try {
      const response = await fetch('/api/data');
      if (!response.ok) throw new Error('Request failed');

      const data = await response.json();
      this.emit({ status: 'success', data });
    } catch (error) {
      this.emit({
        status: 'error',
        error: error instanceof Error ? error.message : 'Unknown error',
      });
    }
  };

  reset = () => {
    this.emit({ status: 'idle' });
  };
}
```

### Usage in React

```tsx
function DataDisplay() {
  const [state, cubit] = useBloc(DataFetchCubit);

  return (
    <div>
      <button onClick={cubit.fetchData} disabled={state.status === 'loading'}>
        {state.status === 'loading' ? 'Loading...' : 'Fetch Data'}
      </button>

      {state.status === 'idle' && <p>Click button to load data</p>}
      {state.status === 'loading' && <Spinner />}
      {state.status === 'success' && <p>Data: {state.data}</p>}
      {state.status === 'error' && <p>{state.error}</p>}
    </div>
  );
}
```

**Key benefits**:
- TypeScript prevents accessing `data` when it doesn't exist
- No impossible states (can't be loading AND error)
- Clear, exhaustive handling of all states
- Guard clause prevents duplicate requests

## Retry with Exponential Backoff

Network requests fail. Exponential backoff is the industry-standard retry pattern that:
- Gives requests a chance to succeed
- Doesn't overwhelm struggling servers
- Backs off progressively: 2s → 4s → 8s → 16s

### Implementation

```typescript
interface RetryState {
  status: 'idle' | 'loading' | 'success' | 'error';
  data?: string;
  error?: string;
  retryCount: number;
  nextRetryIn?: number; // Countdown in seconds
}

class RetryFetchCubit extends Cubit<RetryState> {
  private retryTimer: NodeJS.Timeout | null = null;
  private countdownTimer: NodeJS.Timeout | null = null;
  private maxRetries = 3;

  constructor() {
    super({ status: 'idle', retryCount: 0 });
  }

  fetchData = async (isRetry = false) => {
    if (this.state.status === 'loading') return;

    this.emit({ ...this.state, status: 'loading', nextRetryIn: undefined });

    try {
      const response = await fetch('/api/data');
      if (!response.ok) throw new Error('Request failed');

      const data = await response.json();
      this.emit({ status: 'success', data, retryCount: this.state.retryCount });
    } catch (error) {
      const newRetryCount = isRetry ? this.state.retryCount : this.state.retryCount + 1;

      if (newRetryCount >= this.maxRetries) {
        // Max retries reached - give up
        this.emit({
          status: 'error',
          error: `Failed after ${this.maxRetries} attempts`,
          retryCount: newRetryCount,
        });
      } else {
        // Calculate exponential backoff: 2^retryCount seconds
        const delaySeconds = Math.pow(2, newRetryCount);

        this.emit({
          status: 'error',
          error: error instanceof Error ? error.message : 'Unknown error',
          retryCount: newRetryCount,
          nextRetryIn: delaySeconds,
        });

        this.scheduleRetry(delaySeconds);
      }
    }
  };

  private scheduleRetry = (delaySeconds: number) => {
    // Clear any existing timers
    if (this.retryTimer) clearTimeout(this.retryTimer);
    if (this.countdownTimer) clearInterval(this.countdownTimer);

    // Countdown timer (for UI feedback)
    let remaining = delaySeconds;
    this.countdownTimer = setInterval(() => {
      remaining--;
      if (remaining > 0) {
        this.patch({ nextRetryIn: remaining });
      } else {
        if (this.countdownTimer) clearInterval(this.countdownTimer);
      }
    }, 1000);

    // Retry timer
    this.retryTimer = setTimeout(() => {
      this.fetchData(true); // isRetry=true to not increment count again
    }, delaySeconds * 1000);
  };

  reset = () => {
    if (this.retryTimer) clearTimeout(this.retryTimer);
    if (this.countdownTimer) clearInterval(this.countdownTimer);
    this.emit({ status: 'idle', retryCount: 0 });
  };

  onDispose = () => {
    // Critical: Clean up timers to prevent memory leaks
    if (this.retryTimer) clearTimeout(this.retryTimer);
    if (this.countdownTimer) clearInterval(this.countdownTimer);
  };
}
```

<GuideArticleNote title="Production Enhancements">
For production apps, add:
- **Jitter**: Add random delay (±20%) to prevent thundering herd
- **Max delay cap**: Limit backoff to ~60 seconds
- **Cancel button**: Let users abort retry sequence
- **Network detection**: Pause retries if offline
</GuideArticleNote>

### Exponential Backoff Explained

| Attempt | Delay | Total Time Elapsed |
|---------|-------|-------------------|
| 1st | Immediate | 0s |
| 2nd | 2s (2^1) | 2s |
| 3rd | 4s (2^2) | 6s |
| 4th | 8s (2^3) | 14s |

This gives failing services time to recover without overwhelming them.

## Optimistic Updates

Optimistic updates provide **instant feedback** by updating the UI immediately, then confirming with the server. If the request fails, revert the change.

### When to Use

**Good for**:
- Liking/favoriting (high success rate)
- Toggling checkboxes
- Adding items to lists
- Simple CRUD operations

**Avoid for**:
- Financial transactions (must be certain)
- Irreversible actions (sending emails, publishing)
- Complex validation (high failure rate)

### Implementation

```typescript
interface TodoItem {
  id: string;
  text: string;
  completed: boolean;
  optimistic?: boolean; // Flag for pending confirmation
  error?: boolean; // Flag for failed operations
}

interface TodoState {
  items: TodoItem[];
  isAdding: boolean;
}

class OptimisticTodoCubit extends Cubit<TodoState> {
  constructor() {
    super({ items: [], isAdding: false });
  }

  addTodo = async (text: string) => {
    const optimisticId = `temp-${Date.now()}`;

    // Step 1: Optimistic update - add immediately with temp ID
    this.patch({
      items: [
        ...this.state.items,
        {
          id: optimisticId,
          text,
          completed: false,
          optimistic: true, // Mark as pending
        },
      ],
      isAdding: true,
    });

    try {
      // Step 2: Send to server
      const response = await fetch('/api/todos', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ text }),
      });

      if (!response.ok) throw new Error('Failed to add todo');

      const { id: realId } = await response.json();

      // Step 3: Success - replace temp ID with real one
      this.patch({
        items: this.state.items.map((item) =>
          item.id === optimisticId
            ? { ...item, id: realId, optimistic: false }
            : item
        ),
        isAdding: false,
      });
    } catch (error) {
      // Step 4: Failure - mark as error (or remove)
      this.patch({
        items: this.state.items.map((item) =>
          item.id === optimisticId
            ? { ...item, error: true, optimistic: false }
            : item
        ),
        isAdding: false,
      });
    }
  };

  toggleTodo = async (id: string) => {
    const item = this.state.items.find((i) => i.id === id);
    if (!item) return;

    // Store original state for rollback
    const originalCompleted = item.completed;

    // Optimistic update
    this.patch({
      items: this.state.items.map((i) =>
        i.id === id ? { ...i, completed: !i.completed, optimistic: true } : i
      ),
    });

    try {
      await fetch(`/api/todos/${id}/toggle`, { method: 'PATCH' });

      // Confirm success
      this.patch({
        items: this.state.items.map((i) =>
          i.id === id ? { ...i, optimistic: false } : i
        ),
      });
    } catch (error) {
      // Rollback on failure
      this.patch({
        items: this.state.items.map((i) =>
          i.id === id
            ? { ...i, completed: originalCompleted, optimistic: false, error: true }
            : i
        ),
      });
    }
  };

  removeTodo = (id: string) => {
    this.patch({
      items: this.state.items.filter((item) => item.id !== id),
    });
  };

  clearErrors = () => {
    this.patch({
      items: this.state.items.filter((item) => !item.error),
    });
  };
}
```

## Pattern Comparison

| Pattern | Best For | UX | Complexity |
|---------|----------|-----|-----------|
| **State Machine** | Critical operations, explicit feedback | Clear, honest status | Low |
| **Retry + Backoff** | Unreliable networks, rate limits | Resilient to failures | Medium |
| **Optimistic Updates** | High-success ops, instant feedback | Feels instant | High |

## Common Pitfalls

### Pitfall 1: Not Preventing Duplicate Requests

```typescript
// ❌ BAD: Can trigger multiple simultaneous requests
fetchData = async () => {
  this.emit({ status: 'loading' });
  // ...
};

// ✅ GOOD: Guard against duplicates
fetchData = async () => {
  if (this.state.status === 'loading') return; // Guard!
  this.emit({ status: 'loading' });
  // ...
};
```

### Pitfall 2: Forgetting Cleanup

```typescript
// ❌ BAD: Memory leak - timers never cleared
class CubitWithTimer extends Cubit<State> {
  private timer: NodeJS.Timeout | null = null;

  startRetry = () => {
    this.timer = setTimeout(/* ... */, 5000);
  };
  // No onDispose - timer keeps running after component unmounts!
}

// ✅ GOOD: Clean up timers
class CubitWithTimer extends Cubit<State> {
  private timer: NodeJS.Timeout | null = null;

  onDispose = () => {
    if (this.timer) {
      clearTimeout(this.timer);
      this.timer = null;
    }
  };
}
```

### Pitfall 3: Optimistic Updates for Critical Operations

```typescript
// ❌ BAD: Optimistic update for payment
withdrawMoney = async (amount: number) => {
  // Immediately update balance (what if it fails?!)
  this.patch({ balance: this.state.balance - amount });
  await api.withdraw(amount);
};

// ✅ GOOD: Wait for confirmation on critical operations
withdrawMoney = async (amount: number) => {
  this.patch({ status: 'processing' });
  const result = await api.withdraw(amount);
  this.patch({ balance: result.newBalance, status: 'success' });
};
```

### Pitfall 4: Not Handling Network Offline

```typescript
// ✅ GOOD: Detect offline state
fetchData = async () => {
  if (!navigator.onLine) {
    this.emit({ status: 'error', error: 'No internet connection' });
    return;
  }

  // ... proceed with fetch
};
```

## Best Practices

1. **Use discriminated unions** for type-safe state machines
2. **Guard against duplicate requests** with `if (loading) return;`
3. **Clean up timers** in `onDispose` to prevent memory leaks
4. **Provide user control**: Cancel, retry, and reset actions
5. **Show clear error messages**: Specific, actionable feedback
6. **Use AbortController** for cancellable requests
7. **Implement exponential backoff** for resilient retry logic
8. **Reserve optimistic updates** for high-success operations
9. **Visual feedback** for all async states (loading, error, optimistic)
10. **Test error paths**: Don't just test the happy path

<GuideArticleNote title="AbortController for Cancellation">
Use `AbortController` to cancel in-flight requests when components unmount:

```typescript
class FetchCubit extends Cubit<State> {
  private controller: AbortController | null = null;

  fetchData = async () => {
    // Cancel previous request
    if (this.controller) {
      this.controller.abort();
    }

    this.controller = new AbortController();

    try {
      const response = await fetch('/api/data', {
        signal: this.controller.signal,
      });
      // ...
    } catch (error) {
      if (error.name !== 'AbortError') {
        // Handle real errors (not cancellation)
        this.emit({ status: 'error', error: error.message });
      }
    }
  };

  onDispose = () => {
    if (this.controller) {
      this.controller.abort();
    }
  };
}
```
</GuideArticleNote>

## Testing Async Operations

```typescript
import { describe, test, expect, vi } from 'vitest';

describe('DataFetchCubit', () => {
  test('transitions to loading then success', async () => {
    const cubit = new DataFetchCubit();

    expect(cubit.state).toEqual({ status: 'idle' });

    const fetchPromise = cubit.fetchData();
    expect(cubit.state).toEqual({ status: 'loading' });

    await fetchPromise;
    expect(cubit.state.status).toBe('success');
    expect(cubit.state).toHaveProperty('data');

    cubit.dispose();
  });

  test('prevents duplicate requests', async () => {
    const cubit = new DataFetchCubit();
    const fetchSpy = vi.spyOn(global, 'fetch');

    cubit.fetchData();
    cubit.fetchData(); // Second call should be ignored

    // Wait for fetch to complete
    await new Promise(resolve => setTimeout(resolve, 100));

    expect(fetchSpy).toHaveBeenCalledTimes(1); // Only called once

    cubit.dispose();
  });

  test('retries with backoff on failure', async () => {
    const cubit = new RetryFetchCubit();

    cubit.fetchData();
    await new Promise(resolve => setTimeout(resolve, 100));

    expect(cubit.state.retryCount).toBeGreaterThan(0);
    expect(cubit.state.nextRetryIn).toBeGreaterThan(0);

    cubit.dispose();
  });
});
```

## Key Takeaways

- **State machine pattern**: Use discriminated unions for type-safe async states
- **Exponential backoff**: Resilient retry logic with 2^n delay (2s, 4s, 8s...)
- **Optimistic updates**: Instant feedback for high-success operations
- **Guard clauses**: Prevent duplicate requests with `if (loading) return;`
- **Cleanup timers**: Always implement `onDispose` to clear timeouts/intervals
- **AbortController**: Cancel requests when components unmount
- **User control**: Provide cancel, retry, and reset actions
- **Visual feedback**: Show loading spinners, error states, and pending operations
