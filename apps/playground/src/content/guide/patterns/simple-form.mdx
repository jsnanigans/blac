# Simple Form Handling

Forms are everywhere in modern web applications—from contact forms to search filters to user profiles. Managing form state effectively is crucial for building robust, user-friendly interfaces. Let's explore how BlaC makes form handling straightforward and type-safe.

## What You'll Learn

- Managing form input state with Cubits
- Input binding patterns for controlled components
- Computed properties for validation and submit logic
- Handling form submission with async operations
- Reset and clear patterns
- Testing form logic

## Why Use BlaC for Forms?

Traditional React form handling often leads to:
- **Prop drilling**: Passing handlers through multiple components
- **useState sprawl**: Multiple state hooks for each field
- **Scattered validation**: Logic spread across components
- **Testing challenges**: Hard to test form logic in isolation

BlaC consolidates all form logic in a single, testable Cubit:

```typescript
// ❌ Traditional React: Scattered state
function ContactForm() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');
  const [message, setMessage] = useState('');
  const [isSubmitting, setIsSubmitting] = useState(false);

  // Validation logic mixed with UI
  const canSubmit = name && email && message && !isSubmitting;

  // Submit logic in component
  const handleSubmit = async (e) => {
    e.preventDefault();
    setIsSubmitting(true);
    await fetch('/api/contact', { /* ... */ });
    setIsSubmitting(false);
  };

  // Component becomes very long...
}

// ✅ BlaC: Centralized, testable logic
class ContactFormCubit extends Cubit<ContactFormState> {
  // All form logic in one place
  get canSubmit() { /* ... */ }
  updateName = (name: string) => { /* ... */ };
  submit = async () => { /* ... */ };
}

function ContactForm() {
  const [state, cubit] = useBloc(ContactFormCubit);
  // Clean, focused UI code
}
```

## Basic Form Pattern

Let's build a simple contact form step by step.

### Step 1: Define State Interface

Start with a **flat state structure** for optimal dependency tracking:

```typescript
interface ContactFormState {
  name: string;
  email: string;
  message: string;
  isSubmitting: boolean;
  submitCount: number;
}
```

<GuideArticleNote title="Always Use Flat State">
Use root-level properties like `emailValue`, `emailError`, `emailTouched` instead of nested objects like `email: { value, error, touched }`.

BlaC's dependency tracking only works at the **root level**. Nested properties won't trigger re-renders correctly.
</GuideArticleNote>

### Step 2: Create Form Cubit

Implement the Cubit with update methods and computed properties:

```typescript
import { Cubit } from '@blac/core';

class ContactFormCubit extends Cubit<ContactFormState> {
  constructor() {
    super({
      name: '',
      email: '',
      message: '',
      isSubmitting: false,
      submitCount: 0,
    });
  }

  // Computed property: Is form empty?
  get isEmpty(): boolean {
    return !this.state.name &&
           !this.state.email &&
           !this.state.message;
  }

  // Computed property: Can we submit?
  get canSubmit(): boolean {
    return this.state.name.length > 0 &&
           this.state.email.length > 0 &&
           this.state.message.length > 0 &&
           !this.state.isSubmitting;
  }

  // Update methods (arrow functions!)
  updateName = (name: string) => {
    this.patch({ name });
  };

  updateEmail = (email: string) => {
    this.patch({ email });
  };

  updateMessage = (message: string) => {
    this.patch({ message });
  };

  // Submit handler
  submit = async () => {
    if (!this.canSubmit) return;

    this.patch({ isSubmitting: true });

    try {
      await fetch('/api/contact', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          name: this.state.name,
          email: this.state.email,
          message: this.state.message,
        }),
      });

      this.patch({
        isSubmitting: false,
        submitCount: this.state.submitCount + 1,
      });
    } catch (error) {
      console.error('Submit failed:', error);
      this.patch({ isSubmitting: false });
    }
  };

  // Reset form to initial state
  reset = () => {
    this.emit({
      name: '',
      email: '',
      message: '',
      isSubmitting: false,
      submitCount: this.state.submitCount, // Preserve count
    });
  };
}
```

### Step 3: Connect to React

Use `useBloc` to connect your form to the Cubit:

```tsx
import { useBloc } from '@blac/react';

function ContactForm() {
  const [state, cubit] = useBloc(ContactFormCubit);

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    await cubit.submit();
  };

  return (
    <form onSubmit={handleSubmit}>
      <div>
        <label>Name</label>
        <input
          type="text"
          value={state.name}
          onChange={(e) => cubit.updateName(e.target.value)}
          placeholder="John Doe"
        />
      </div>

      <div>
        <label>Email</label>
        <input
          type="email"
          value={state.email}
          onChange={(e) => cubit.updateEmail(e.target.value)}
          placeholder="john@example.com"
        />
      </div>

      <div>
        <label>Message</label>
        <textarea
          value={state.message}
          onChange={(e) => cubit.updateMessage(e.target.value)}
          rows={4}
          placeholder="Your message here..."
        />
      </div>

      <div>
        <button
          type="submit"
          disabled={!cubit.canSubmit}
        >
          {state.isSubmitting ? 'Submitting...' : 'Submit'}
        </button>
        <button
          type="button"
          onClick={() => cubit.reset()}
          disabled={cubit.isEmpty}
        >
          Reset
        </button>
      </div>

      {state.submitCount > 0 && !state.isSubmitting && (
        <div>
          Form submitted successfully! (Count: {state.submitCount})
        </div>
      )}
    </form>
  );
}
```

<GuideArticleNote title="Controlled Components">
BlaC forms use **controlled components**: the Cubit state is the single source of truth. Input values always reflect `state.fieldName`, and changes flow through update methods.

This ensures predictable behavior and makes testing straightforward.
</GuideArticleNote>

## Input Binding Patterns

### Pattern 1: Direct Binding

The simplest approach—one update method per field:

```typescript
class FormCubit extends Cubit<FormState> {
  updateEmail = (email: string) => {
    this.patch({ email });
  };

  updatePassword = (password: string) => {
    this.patch({ password });
  };
}

// Usage
<input
  value={state.email}
  onChange={(e) => cubit.updateEmail(e.target.value)}
/>
```

**Pros**: Clear, explicit, type-safe
**Cons**: Repetitive for large forms

### Pattern 2: Generic Update Method

Use a generic method with field name:

```typescript
class FormCubit extends Cubit<FormState> {
  updateField = (field: keyof FormState, value: string) => {
    this.patch({ [field]: value } as Partial<FormState>);
  };
}

// Usage
<input
  value={state.email}
  onChange={(e) => cubit.updateField('email', e.target.value)}
/>
```

**Pros**: Less code, DRY
**Cons**: Loses some type safety, less explicit

### Pattern 3: Event-Based (for complex forms)

For forms with validation, use name attributes:

```typescript
class FormCubit extends Cubit<FormState> {
  handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const { name, value } = e.target;
    this.patch({ [name]: value } as Partial<FormState>);
  };
}

// Usage
<input
  name="email"
  value={state.email}
  onChange={cubit.handleChange}
/>
```

**Pros**: Minimal JSX, familiar pattern
**Cons**: Relies on name attributes, less type-safe

**Recommendation**: Use **Pattern 1 (Direct Binding)** for most forms. It's the most explicit and type-safe.

## Computed Properties for Validation

Computed properties (getters) are perfect for form validation:

```typescript
class LoginFormCubit extends Cubit<LoginFormState> {
  // Field-level validation
  get emailError(): string | null {
    if (!this.state.emailTouched) return null;
    if (!this.state.emailValue) return 'Email is required';
    if (!this.state.emailValue.includes('@')) return 'Invalid email';
    return null;
  }

  get passwordError(): string | null {
    if (!this.state.passwordTouched) return null;
    if (!this.state.passwordValue) return 'Password is required';
    if (this.state.passwordValue.length < 8) return 'Min 8 characters';
    return null;
  }

  // Form-level validation
  get isValid(): boolean {
    return this.emailError === null &&
           this.passwordError === null &&
           this.state.emailTouched &&
           this.state.passwordTouched;
  }

  get canSubmit(): boolean {
    return this.isValid && !this.state.isSubmitting;
  }
}

// Usage in component
<input
  value={state.emailValue}
  onChange={(e) => cubit.updateEmail(e.target.value)}
  className={cubit.emailError ? 'error' : ''}
/>
{cubit.emailError && <span>{cubit.emailError}</span>}
```

<GuideArticleNote title="Flat State for Validation">
Store validation data as **separate root-level properties**:

```typescript
// ✅ GOOD: Flat state
interface FormState {
  emailValue: string;
  emailError: string | null;
  emailTouched: boolean;
}

// ❌ BAD: Nested state
interface FormState {
  email: {
    value: string;
    error: string | null;
    touched: boolean;
  }
}
```

Flat state ensures dependency tracking works correctly.
</GuideArticleNote>

## Handling Form Submission

### Basic Submit Pattern

```typescript
class FormCubit extends Cubit<FormState> {
  submit = async () => {
    // Guard: Check if submission is allowed
    if (!this.canSubmit) return;

    // Set loading state
    this.patch({ isSubmitting: true });

    try {
      // API call
      const response = await fetch('/api/submit', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          email: this.state.email,
          password: this.state.password,
        }),
      });

      if (!response.ok) {
        throw new Error('Submit failed');
      }

      // Success: Update state
      this.patch({
        isSubmitting: false,
        submitCount: this.state.submitCount + 1,
      });
    } catch (error) {
      // Error: Update state
      this.patch({
        isSubmitting: false,
        submitError: error.message,
      });
    }
  };
}
```

### Submit with Validation

```typescript
class FormCubit extends Cubit<FormState> {
  submit = async () => {
    // Mark all fields as touched (show all errors)
    this.patch({
      emailTouched: true,
      passwordTouched: true,
    });

    // Check validation
    if (!this.isValid) {
      return; // Don't submit if invalid
    }

    // Proceed with submission
    this.patch({ isSubmitting: true });
    // ... rest of submit logic
  };
}
```

### Optimistic Updates

For better UX, update UI immediately and rollback on error:

```typescript
class FormCubit extends Cubit<FormState> {
  submit = async () => {
    // Store current state for rollback
    const previousState = { ...this.state };

    // Optimistic update
    this.patch({
      isSubmitting: true,
      submitCount: this.state.submitCount + 1,
    });

    try {
      await fetch('/api/submit', { /* ... */ });
      // Success: Keep optimistic state
      this.patch({ isSubmitting: false });
    } catch (error) {
      // Rollback on error
      this.emit(previousState);
      this.patch({ submitError: error.message });
    }
  };
}
```

## Reset and Clear Patterns

### Pattern 1: Full Reset

Reset all fields to initial state:

```typescript
class FormCubit extends Cubit<FormState> {
  reset = () => {
    this.emit({
      name: '',
      email: '',
      message: '',
      isSubmitting: false,
      submitCount: this.state.submitCount, // Preserve if needed
    });
  };
}
```

### Pattern 2: Partial Reset

Clear only form fields, preserve metadata:

```typescript
class FormCubit extends Cubit<FormState> {
  clearFields = () => {
    this.patch({
      name: '',
      email: '',
      message: '',
      // Keep isSubmitting, submitCount, etc.
    });
  };
}
```

### Pattern 3: Smart Reset

Reset with preserved values:

```typescript
class FormCubit extends Cubit<FormState> {
  private initialState: FormState;

  constructor() {
    super({
      name: '',
      email: '',
      message: '',
      isSubmitting: false,
      submitCount: 0,
    });
    this.initialState = { ...this.state };
  }

  reset = () => {
    this.emit({
      ...this.initialState,
      submitCount: this.state.submitCount, // Preserve count
    });
  };
}
```

## Real-World Example: Newsletter Form

A more complex form with multiple input types:

```typescript
interface NewsletterState {
  email: string;
  frequency: 'daily' | 'weekly' | 'monthly';
  interests: string[];
  agreeToTerms: boolean;
  isSubmitting: boolean;
  submitCount: number;
}

class NewsletterCubit extends Cubit<NewsletterState> {
  constructor() {
    super({
      email: '',
      frequency: 'weekly',
      interests: [],
      agreeToTerms: false,
      isSubmitting: false,
      submitCount: 0,
    });
  }

  get canSubmit(): boolean {
    return this.state.email.includes('@') &&
           this.state.agreeToTerms &&
           !this.state.isSubmitting;
  }

  updateEmail = (email: string) => {
    this.patch({ email });
  };

  updateFrequency = (frequency: 'daily' | 'weekly' | 'monthly') => {
    this.patch({ frequency });
  };

  toggleInterest = (interest: string) => {
    const interests = this.state.interests.includes(interest)
      ? this.state.interests.filter(i => i !== interest)
      : [...this.state.interests, interest];
    this.patch({ interests });
  };

  toggleTerms = () => {
    this.patch({ agreeToTerms: !this.state.agreeToTerms });
  };

  submit = async () => {
    if (!this.canSubmit) return;

    this.patch({ isSubmitting: true });

    try {
      await fetch('/api/newsletter', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          email: this.state.email,
          frequency: this.state.frequency,
          interests: this.state.interests,
        }),
      });

      this.patch({
        isSubmitting: false,
        submitCount: this.state.submitCount + 1,
      });
    } catch (error) {
      this.patch({ isSubmitting: false });
    }
  };
}
```

## Testing Form Logic

BlaC makes form testing straightforward—test the Cubit in isolation:

```typescript
import { describe, test, expect } from 'vitest';

describe('ContactFormCubit', () => {
  test('starts with empty fields', () => {
    const cubit = new ContactFormCubit();

    expect(cubit.state.name).toBe('');
    expect(cubit.state.email).toBe('');
    expect(cubit.state.message).toBe('');
    expect(cubit.isEmpty).toBe(true);

    cubit.dispose();
  });

  test('updates individual fields', () => {
    const cubit = new ContactFormCubit();

    cubit.updateName('John');
    expect(cubit.state.name).toBe('John');

    cubit.updateEmail('john@example.com');
    expect(cubit.state.email).toBe('john@example.com');

    cubit.dispose();
  });

  test('canSubmit requires all fields', () => {
    const cubit = new ContactFormCubit();

    expect(cubit.canSubmit).toBe(false);

    cubit.updateName('John');
    expect(cubit.canSubmit).toBe(false); // Still false

    cubit.updateEmail('john@example.com');
    expect(cubit.canSubmit).toBe(false); // Still false

    cubit.updateMessage('Hello');
    expect(cubit.canSubmit).toBe(true); // Now true!

    cubit.dispose();
  });

  test('submit sets loading state', async () => {
    const cubit = new ContactFormCubit();

    cubit.updateName('John');
    cubit.updateEmail('john@example.com');
    cubit.updateMessage('Hello');

    const submitPromise = cubit.submit();
    expect(cubit.state.isSubmitting).toBe(true);
    expect(cubit.canSubmit).toBe(false); // Can't submit while submitting

    await submitPromise;
    expect(cubit.state.isSubmitting).toBe(false);

    cubit.dispose();
  });

  test('reset clears all fields', () => {
    const cubit = new ContactFormCubit();

    cubit.updateName('John');
    cubit.updateEmail('john@example.com');
    cubit.updateMessage('Hello');

    cubit.reset();

    expect(cubit.state.name).toBe('');
    expect(cubit.state.email).toBe('');
    expect(cubit.state.message).toBe('');
    expect(cubit.isEmpty).toBe(true);

    cubit.dispose();
  });
});
```

## Common Pitfalls

### Pitfall 1: Nested State

```typescript
// ❌ BAD: Nested state breaks dependency tracking
interface FormState {
  fields: {
    email: string;
    password: string;
  };
  meta: {
    isSubmitting: boolean;
  };
}

// ✅ GOOD: Flat state
interface FormState {
  email: string;
  password: string;
  isSubmitting: boolean;
}
```

### Pitfall 2: Forgetting Arrow Functions

```typescript
// ❌ BAD: Regular method loses `this` context
class FormCubit extends Cubit<FormState> {
  updateEmail(email: string) { // Will break!
    this.patch({ email });
  }
}

// ✅ GOOD: Arrow function preserves `this`
class FormCubit extends Cubit<FormState> {
  updateEmail = (email: string) => {
    this.patch({ email });
  };
}
```

### Pitfall 3: Not Preventing Double Submit

```typescript
// ❌ BAD: Can submit multiple times
submit = async () => {
  this.patch({ isSubmitting: true });
  await fetch('/api/submit', { /* ... */ });
  this.patch({ isSubmitting: false });
};

// ✅ GOOD: Guard against double submit
submit = async () => {
  if (!this.canSubmit) return; // Guard!
  this.patch({ isSubmitting: true });
  await fetch('/api/submit', { /* ... */ });
  this.patch({ isSubmitting: false });
};
```

### Pitfall 4: Mutating Arrays/Objects

```typescript
// ❌ BAD: Mutates state array
toggleInterest = (interest: string) => {
  this.state.interests.push(interest); // Mutation!
  this.patch({ interests: this.state.interests });
};

// ✅ GOOD: Creates new array
toggleInterest = (interest: string) => {
  const interests = [...this.state.interests, interest];
  this.patch({ interests });
};
```

## Best Practices

1. **Use Flat State**: Always use root-level properties for dependency tracking
2. **Arrow Functions**: All methods must be arrow functions
3. **Computed Properties**: Use getters for derived values like `canSubmit` and `isValid`
4. **Guard Submit**: Always check `if (!this.canSubmit) return;` at the start of submit
5. **Loading States**: Track `isSubmitting` to disable buttons and show loading UI
6. **Type Safety**: Define strict TypeScript interfaces for state
7. **Test in Isolation**: Test Cubit logic without rendering React components
8. **Single Responsibility**: One Cubit per form, keep logic focused

## Key Takeaways

- **Centralized Logic**: Forms become testable, reusable Cubits
- **Controlled Components**: State flows one way—from Cubit to UI
- **Flat State**: Root-level properties for optimal dependency tracking
- **Computed Properties**: Getters provide derived values like validation
- **Type Safety**: TypeScript ensures correctness
- **Testability**: Test form logic in isolation from UI
