# Data Fetching & Caching

## Introduction

Fetching data from APIs is one of the most common tasks in web development. The difference between a **naive implementation** that makes redundant requests and a **production-ready solution** with smart caching can dramatically impact both user experience and server load.

In this guide, you'll learn three battle-tested data fetching patterns used by companies like Vercel, Airbnb, and Netflix to build fast, responsive applications.

### What You'll Learn

- Implement time-based caching to avoid redundant API calls
- Build a Stale-While-Revalidate (SWR) pattern for instant perceived performance
- Handle large datasets efficiently with pagination
- Prevent race conditions and memory leaks in async operations
- Test data fetching logic comprehensively

## The Problem: Naive Data Fetching

Let's start by understanding what **not** to do:

```typescript
// ❌ NAIVE: Fetches every time, no caching
class NaiveCubit extends Cubit<{ data: User[] | null; loading: boolean }> {
  fetchUsers = async () => {
    this.patch({ loading: true });
    const users = await api.getUsers(); // Always hits the server
    this.patch({ data: users, loading: false });
  };
}
```

**Problems with this approach:**
1. **Redundant requests**: Fetching the same data repeatedly wastes bandwidth
2. **Poor UX**: Users see loading spinners even when data hasn't changed
3. **Server load**: Your backend gets hammered with duplicate requests
4. **Cost**: More API calls = higher cloud bills

The solution? **Smart caching strategies.**

## Pattern 1: Simple Time-Based Cache

The simplest effective pattern: if we fetched data recently (within X seconds), reuse it. Otherwise, fetch fresh data.

```typescript
interface CachedState<T> {
  data: T | null;
  loading: boolean;
  error: string | null;
  cachedAt: number | null; // Timestamp when data was cached
}

class SimpleCacheCubit extends Cubit<CachedState<User[]>> {
  private cacheTimeout = 5000; // 5 seconds

  constructor() {
    super({
      data: null,
      loading: false,
      error: null,
      cachedAt: null,
    });
  }

  fetchUsers = async () => {
    // ✅ CRITICAL: Check if cache is still fresh
    if (this.isCacheFresh()) {
      console.log('[Cache] Using cached data');
      return; // Skip API call entirely
    }

    this.patch({ loading: true, error: null });

    try {
      const users = await api.getUsers();
      this.patch({
        data: users,
        loading: false,
        cachedAt: Date.now(), // Store cache timestamp
      });
    } catch (error) {
      this.patch({
        loading: false,
        error: error instanceof Error ? error.message : 'Failed to fetch',
      });
    }
  };

  private isCacheFresh = (): boolean => {
    if (!this.state.cachedAt || !this.state.data) return false;
    const age = Date.now() - this.state.cachedAt;
    return age < this.cacheTimeout;
  };

  // Computed property for displaying cache age
  get cacheAge(): number | null {
    if (!this.state.cachedAt) return null;
    return Math.floor((Date.now() - this.state.cachedAt) / 1000);
  }

  // Manual invalidation when you know data is stale
  invalidateCache = () => {
    this.patch({ cachedAt: null });
  };
}
```

### Using Simple Cache in React

```tsx
function UserList() {
  const [state, cubit] = useBloc(SimpleCacheCubit);

  return (
    <div>
      <button onClick={cubit.fetchUsers} disabled={state.loading}>
        {state.loading ? 'Loading...' : 'Fetch Users'}
      </button>

      {state.cachedAt && (
        <p>Cached {cubit.cacheAge}s ago</p>
      )}

      {state.data?.map(user => (
        <div key={user.id}>{user.name}</div>
      ))}
    </div>
  );
}
```

<GuideArticleNote title="When to Use Simple Caching">
Simple time-based caching works best for:

- **Infrequently changing data**: User profiles, app settings, configuration
- **Non-critical freshness**: Slight staleness won't impact user experience
- **Reducing server load**: During rapid page navigation or component re-renders
- **Cost optimization**: Especially important with metered APIs

**Don't use** for real-time or frequently changing data (stock prices, live scores, etc.).
</GuideArticleNote>

## Pattern 2: Stale-While-Revalidate (SWR)

SWR is the secret sauce behind fast, responsive apps like Twitter and GitHub. The strategy:

1. **Return cached data immediately** (even if stale)
2. **Fetch fresh data in the background**
3. **Update UI when fresh data arrives**

This gives users **instant feedback** while ensuring they eventually see fresh data.

```typescript
interface SWRState {
  data: Article[] | null;
  loading: boolean;
  revalidating: boolean; // Background refresh indicator
  error: string | null;
  cachedAt: number | null;
}

class SWRCubit extends Cubit<SWRState> {
  private cacheTimeout = 8000; // 8 seconds
  private autoRefreshInterval: NodeJS.Timeout | null = null;

  constructor() {
    super({
      data: null,
      loading: false,
      revalidating: false,
      error: null,
      cachedAt: null,
    });
  }

  fetchArticles = async (background = false) => {
    // ✅ KEY DIFFERENCE: Show cached data, revalidate in background
    if (this.state.data && !background) {
      // We have data! Show it immediately
      this.patch({ revalidating: true }); // Subtle background indicator
    } else if (!background) {
      // First load: show loading spinner
      this.patch({ loading: true, error: null });
    }

    try {
      const articles = await api.getArticles();
      this.patch({
        data: articles,
        loading: false,
        revalidating: false,
        cachedAt: Date.now(),
      });
    } catch (error) {
      this.patch({
        loading: false,
        revalidating: false,
        error: error instanceof Error ? error.message : 'Failed to fetch',
      });
    }
  };

  // Optional: Auto-refresh data every X seconds
  enableAutoRefresh = () => {
    this.disableAutoRefresh(); // Clear existing timer
    this.autoRefreshInterval = setInterval(() => {
      console.log('[SWR] Auto-refreshing...');
      this.fetchArticles(true); // Background refresh
    }, this.cacheTimeout);
  };

  disableAutoRefresh = () => {
    if (this.autoRefreshInterval) {
      clearInterval(this.autoRefreshInterval);
      this.autoRefreshInterval = null;
    }
  };

  get cacheAge(): number | null {
    if (!this.state.cachedAt) return null;
    return Math.floor((Date.now() - this.state.cachedAt) / 1000);
  }

  // ✅ CRITICAL: Clean up timers on disposal
  onDispose = () => {
    this.disableAutoRefresh();
  };
}
```

### Using SWR Pattern in React

```tsx
function ArticleList() {
  const [state, cubit] = useBloc(SWRCubit);

  return (
    <div>
      <button onClick={() => cubit.fetchArticles()}>
        Refresh
      </button>

      {/* Show subtle revalidation indicator */}
      {state.revalidating && (
        <span>↻ Updating...</span>
      )}

      {/* Data appears instantly on subsequent visits */}
      {state.data?.map(article => (
        <article key={article.id}>
          <h2>{article.title}</h2>
          <p>By {article.author}</p>
        </article>
      ))}
    </div>
  );
}
```

<GuideArticleNote title="SWR vs Simple Cache">
**Simple Cache** waits for cache expiry before fetching fresh data.

**SWR** shows stale data immediately and fetches fresh data in parallel.

**Choose SWR when**:
- User experience depends on perceived speed
- Data changes moderately often (every few seconds/minutes)
- You can tolerate showing slightly stale data briefly
- You want automatic background refreshing
</GuideArticleNote>

## Pattern 3: Pagination

When dealing with large datasets (hundreds or thousands of items), loading everything at once is **slow**, **wasteful**, and **bad UX**. Pagination loads data in manageable chunks.

```typescript
interface PaginatedState {
  data: Product[];
  currentPage: number;
  totalPages: number;
  loading: boolean;
  error: string | null;
}

class PaginationCubit extends Cubit<PaginatedState> {
  private pageSize = 20;

  constructor() {
    super({
      data: [],
      currentPage: 1,
      totalPages: 0,
      loading: false,
      error: null,
    });
  }

  loadPage = async (page: number) => {
    // Guard: Don't load invalid pages
    if (this.state.loading) return;

    this.patch({ loading: true, error: null });

    try {
      // API request with pagination parameters
      const response = await api.getProducts({
        page,
        pageSize: this.pageSize,
      });

      this.patch({
        data: response.data,
        currentPage: page,
        totalPages: response.totalPages,
        loading: false,
      });
    } catch (error) {
      this.patch({
        loading: false,
        error: error instanceof Error ? error.message : 'Failed to load page',
      });
    }
  };

  nextPage = () => {
    if (this.state.currentPage < this.state.totalPages && !this.state.loading) {
      this.loadPage(this.state.currentPage + 1);
    }
  };

  prevPage = () => {
    if (this.state.currentPage > 1 && !this.state.loading) {
      this.loadPage(this.state.currentPage - 1);
    }
  };

  goToPage = (page: number) => {
    if (page >= 1 && page <= this.state.totalPages && !this.state.loading) {
      this.loadPage(page);
    }
  };
}
```

### Using Pagination in React

```tsx
function ProductCatalog() {
  const [state, cubit] = useBloc(PaginationCubit);

  // Load first page on mount
  useEffect(() => {
    cubit.loadPage(1);
  }, []);

  return (
    <div>
      {/* Product grid */}
      <div>
        {state.data.map(product => (
          <ProductCard key={product.id} product={product} />
        ))}
      </div>

      {/* Pagination controls */}
      <div>
        <button
          onClick={cubit.prevPage}
          disabled={state.currentPage === 1 || state.loading}
        >
          Previous
        </button>

        <span>
          Page {state.currentPage} of {state.totalPages}
        </span>

        <button
          onClick={cubit.nextPage}
          disabled={state.currentPage === state.totalPages || state.loading}
        >
          Next
        </button>
      </div>
    </div>
  );
}
```

## Advanced Considerations

### 1. Cache Invalidation Strategies

There are two hard problems in computer science: naming things and cache invalidation. Here's how to handle the latter:

```typescript
class SmartCacheCubit extends Cubit<CachedState<User[]>> {
  // Time-based invalidation
  private isCacheStale = (): boolean => {
    if (!this.state.cachedAt) return true;
    const age = Date.now() - this.state.cachedAt;
    return age > this.cacheTimeout;
  };

  // Manual invalidation after mutations
  updateUser = async (userId: string, updates: Partial<User>) => {
    await api.updateUser(userId, updates);
    this.invalidateCache(); // Force refetch on next access
    this.fetchUsers(); // Or refetch immediately
  };

  // Event-based invalidation
  onUserLogout = () => {
    this.clearCache(); // Clear sensitive data
  };

  private clearCache = () => {
    this.patch({ data: null, cachedAt: null });
  };
}
```

### 2. Preventing Race Conditions

When multiple requests are in flight, later requests might resolve before earlier ones, causing stale data to overwrite fresh data:

```typescript
class SafeFetchCubit extends Cubit<State> {
  private currentRequestId = 0;

  fetchData = async () => {
    const requestId = ++this.currentRequestId;

    this.patch({ loading: true });

    try {
      const data = await api.getData(); // Slow request

      // ✅ CRITICAL: Only update if this is still the latest request
      if (requestId === this.currentRequestId) {
        this.patch({ data, loading: false });
      } else {
        console.log('[Race] Discarding stale response');
      }
    } catch (error) {
      if (requestId === this.currentRequestId) {
        this.patch({ error: error.message, loading: false });
      }
    }
  };
}
```

### 3. Request Deduplication

Prevent duplicate simultaneous requests to the same endpoint:

```typescript
class DeduplicatedCubit extends Cubit<State> {
  private activeRequest: Promise<Data> | null = null;

  fetchData = async (): Promise<Data> => {
    // If request is already in flight, return existing promise
    if (this.activeRequest) {
      console.log('[Dedupe] Reusing active request');
      return this.activeRequest;
    }

    this.patch({ loading: true });

    this.activeRequest = api.getData()
      .then(data => {
        this.patch({ data, loading: false });
        this.activeRequest = null;
        return data;
      })
      .catch(error => {
        this.patch({ error: error.message, loading: false });
        this.activeRequest = null;
        throw error;
      });

    return this.activeRequest;
  };
}
```

<GuideArticleNote title="Memory Leaks Warning">
**Always** clean up timers and AbortControllers in `onDispose`:

```typescript
class SafeCubit extends Cubit<State> {
  private refreshTimer: NodeJS.Timeout | null = null;
  private abortController: AbortController | null = null;

  fetchData = async () => {
    this.abortController = new AbortController();

    try {
      const data = await api.getData({
        signal: this.abortController.signal
      });
      this.patch({ data });
    } catch (error) {
      if (error.name !== 'AbortError') {
        this.patch({ error: error.message });
      }
    }
  };

  onDispose = () => {
    if (this.refreshTimer) clearTimeout(this.refreshTimer);
    if (this.abortController) this.abortController.abort();
  };
}
```

Failing to clean up will cause memory leaks and unnecessary network requests after component unmount.
</GuideArticleNote>

## Real-World Example: User Dashboard

Here's a complete example combining multiple patterns:

```typescript
interface DashboardState {
  // User profile (rarely changes - simple cache)
  userProfile: User | null;
  userProfileCachedAt: number | null;

  // Activity feed (updates frequently - SWR)
  activityFeed: Activity[];
  activityFeedCachedAt: number | null;
  activityRevalidating: boolean;

  // Notifications (large list - pagination)
  notifications: Notification[];
  notificationsPage: number;
  notificationsTotalPages: number;

  loading: boolean;
  error: string | null;
}

class DashboardCubit extends Cubit<DashboardState> {
  private profileCacheTimeout = 60000; // 1 minute
  private activityCacheTimeout = 10000; // 10 seconds
  private autoRefreshTimer: NodeJS.Timeout | null = null;

  constructor() {
    super({
      userProfile: null,
      userProfileCachedAt: null,
      activityFeed: [],
      activityFeedCachedAt: null,
      activityRevalidating: false,
      notifications: [],
      notificationsPage: 1,
      notificationsTotalPages: 0,
      loading: false,
      error: null,
    });
  }

  // Simple cache for user profile
  fetchUserProfile = async (userId: string) => {
    if (this.isProfileCacheFresh()) return;

    this.patch({ loading: true });
    try {
      const profile = await api.getUserProfile(userId);
      this.patch({
        userProfile: profile,
        userProfileCachedAt: Date.now(),
        loading: false,
      });
    } catch (error) {
      this.patch({ error: error.message, loading: false });
    }
  };

  // SWR for activity feed
  fetchActivityFeed = async (background = false) => {
    if (this.state.activityFeed.length > 0 && !background) {
      this.patch({ activityRevalidating: true });
    }

    try {
      const activities = await api.getActivityFeed();
      this.patch({
        activityFeed: activities,
        activityFeedCachedAt: Date.now(),
        activityRevalidating: false,
      });
    } catch (error) {
      this.patch({
        error: error.message,
        activityRevalidating: false,
      });
    }
  };

  // Pagination for notifications
  loadNotificationsPage = async (page: number) => {
    this.patch({ loading: true });
    try {
      const response = await api.getNotifications({ page, pageSize: 20 });
      this.patch({
        notifications: response.data,
        notificationsPage: page,
        notificationsTotalPages: response.totalPages,
        loading: false,
      });
    } catch (error) {
      this.patch({ error: error.message, loading: false });
    }
  };

  // Auto-refresh activity feed
  enableAutoRefresh = () => {
    this.autoRefreshTimer = setInterval(() => {
      this.fetchActivityFeed(true);
    }, this.activityCacheTimeout);
  };

  private isProfileCacheFresh = (): boolean => {
    if (!this.state.userProfileCachedAt) return false;
    const age = Date.now() - this.state.userProfileCachedAt;
    return age < this.profileCacheTimeout;
  };

  onDispose = () => {
    if (this.autoRefreshTimer) clearInterval(this.autoRefreshTimer);
  };
}
```

## Testing Data Fetching Logic

Here's how to test caching and fetching behavior:

```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest';

describe('SimpleCacheCubit', () => {
  let cubit: SimpleCacheCubit;
  let mockApi: { getUsers: ReturnType<typeof vi.fn> };

  beforeEach(() => {
    mockApi = { getUsers: vi.fn() };
    cubit = new SimpleCacheCubit(mockApi);
  });

  it('should fetch data on first call', async () => {
    mockApi.getUsers.mockResolvedValue([{ id: 1, name: 'Alice' }]);

    await cubit.fetchUsers();

    expect(mockApi.getUsers).toHaveBeenCalledTimes(1);
    expect(cubit.state.data).toHaveLength(1);
    expect(cubit.state.cachedAt).toBeTruthy();
  });

  it('should use cache on second call within timeout', async () => {
    mockApi.getUsers.mockResolvedValue([{ id: 1, name: 'Alice' }]);

    await cubit.fetchUsers();
    await cubit.fetchUsers(); // Should use cache

    expect(mockApi.getUsers).toHaveBeenCalledTimes(1); // Only called once
  });

  it('should refetch after cache expires', async () => {
    vi.useFakeTimers();
    mockApi.getUsers.mockResolvedValue([{ id: 1, name: 'Alice' }]);

    await cubit.fetchUsers();

    // Fast-forward time past cache timeout
    vi.advanceTimersByTime(6000); // 6 seconds > 5 second cache

    await cubit.fetchUsers();

    expect(mockApi.getUsers).toHaveBeenCalledTimes(2);
    vi.useRealTimers();
  });

  it('should invalidate cache manually', async () => {
    mockApi.getUsers.mockResolvedValue([{ id: 1, name: 'Alice' }]);

    await cubit.fetchUsers();
    cubit.invalidateCache();
    await cubit.fetchUsers();

    expect(mockApi.getUsers).toHaveBeenCalledTimes(2);
  });
});
```

## Key Takeaways

- **Simple caching** prevents redundant API calls for infrequently changing data
- **Stale-While-Revalidate** gives instant perceived performance by showing cached data while fetching fresh data in the background
- **Pagination** makes large datasets manageable and improves both performance and UX
- **Always clean up** timers and AbortControllers in `onDispose` to prevent memory leaks
- **Prevent race conditions** by tracking request IDs or using AbortController
- **Test caching behavior** thoroughly, including cache expiry and invalidation
- **Choose the right pattern** based on data change frequency and UX requirements
