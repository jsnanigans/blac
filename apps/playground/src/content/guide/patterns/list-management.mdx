# List Management & CRUD

## Introduction

Lists are the bread and butter of front-end development: todo lists, shopping carts, user tables, comment threads, product catalogs, and more. Managing lists well—adding items, removing them, updating them, filtering them—is a **fundamental skill** that separates amateur code from professional applications.

The difference between a simple list that "works" and a production-ready implementation often comes down to handling **immutability**, **performance**, **user experience**, and **edge cases**.

In this guide, you'll master three essential list management patterns used in real-world applications.

### What You'll Learn

- Implement CRUD operations (Create, Read, Update, Delete) with immutable state
- Build event-driven lists with filtering and computed properties
- Handle bulk operations efficiently (select all, delete selected)
- Generate stable IDs and normalize nested data
- Batch multiple updates for optimal performance
- Test list operations comprehensively

## Pattern 1: Simple CRUD with Cubit

For straightforward lists without complex business logic, **Cubit** provides the simplest, most direct approach. Perfect for:
- Todo lists
- Shopping lists
- Basic form field arrays
- Simple task managers

```typescript
interface Task {
  id: number;
  text: string;
  completed: boolean;
}

interface SimpleListState {
  tasks: Task[];
  nextId: number; // Auto-incrementing ID
}

class SimpleListCubit extends Cubit<SimpleListState> {
  constructor() {
    super({
      tasks: [
        { id: 1, text: 'Learn list management', completed: false },
        { id: 2, text: 'Build a task list', completed: false },
      ],
      nextId: 3,
    });
  }

  // CREATE: Add a new task
  addTask = (text: string) => {
    if (!text.trim()) return; // Guard clause

    this.patch({
      tasks: [
        ...this.state.tasks, // Spread existing tasks
        { id: this.state.nextId, text: text.trim(), completed: false },
      ],
      nextId: this.state.nextId + 1, // Increment ID
    });
  };

  // UPDATE: Toggle completion status
  toggleTask = (id: number) => {
    this.patch({
      tasks: this.state.tasks.map((task) =>
        task.id === id ? { ...task, completed: !task.completed } : task
      ),
    });
  };

  // DELETE: Remove a task
  removeTask = (id: number) => {
    this.patch({
      tasks: this.state.tasks.filter((task) => task.id !== id),
    });
  };

  // BULK DELETE: Remove all completed tasks
  clearCompleted = () => {
    this.patch({
      tasks: this.state.tasks.filter((task) => !task.completed),
    });
  };
}
```

### Using in React

```tsx
function TaskList() {
  const [state, cubit] = useBloc(SimpleListCubit);
  const [inputValue, setInputValue] = useState('');

  const handleAdd = (e: React.FormEvent) => {
    e.preventDefault();
    if (inputValue.trim()) {
      cubit.addTask(inputValue);
      setInputValue(''); // Clear input
    }
  };

  return (
    <div>
      <form onSubmit={handleAdd}>
        <input
          value={inputValue}
          onChange={(e) => setInputValue(e.target.value)}
          placeholder="Add a task..."
        />
        <button type="submit">Add</button>
      </form>

      {state.tasks.map((task) => (
        <div key={task.id}>
          <input
            type="checkbox"
            checked={task.completed}
            onChange={() => cubit.toggleTask(task.id)}
          />
          <span}>
            {task.text}
          </span>
          <button onClick={() => cubit.removeTask(task.id)}>Delete</button>
        </div>
      ))}

      {state.tasks.filter(t => t.completed).length > 0 && (
        <button onClick={cubit.clearCompleted}>
          Clear Completed
        </button>
      )}
    </div>
  );
}
```

<GuideArticleNote title="Immutability is Critical">
**Never mutate** the original array directly:

```typescript
// ❌ WRONG: Direct mutation
this.state.tasks.push(newTask); // Breaks React's change detection!
this.state.tasks[0].completed = true; // Also wrong!

// ✅ CORRECT: Create new arrays/objects
this.patch({
  tasks: [...this.state.tasks, newTask], // New array
});

this.patch({
  tasks: this.state.tasks.map(task =>
    task.id === id ? { ...task, completed: true } : task // New object
  ),
});
```

React relies on **reference equality** to detect changes. Mutating objects in place means React won't know to re-render.
</GuideArticleNote>

## Pattern 2: Event-Driven Lists with Bloc

For complex lists with **business logic**, **filtering**, **sorting**, or **audit requirements**, use the **Bloc pattern**. The event-driven approach makes state changes explicit and testable.

Use Blocs when you need:
- Multiple filter types (status, priority, category)
- Computed properties derived from list state
- Event logging or undo/redo functionality
- Integration with DevTools for debugging

```typescript
interface Todo {
  id: number;
  text: string;
  completed: boolean;
  priority: 'low' | 'medium' | 'high';
}

interface TodoState {
  todos: Todo[];
  filter: 'all' | 'active' | 'completed';
  priorityFilter: 'all' | 'low' | 'medium' | 'high';
  nextId: number;
}

// Define event classes
class AddTodoEvent {
  constructor(
    public readonly text: string,
    public readonly priority: 'low' | 'medium' | 'high' = 'medium'
  ) {}
}

class ToggleTodoEvent {
  constructor(public readonly id: number) {}
}

class SetFilterEvent {
  constructor(public readonly filter: 'all' | 'active' | 'completed') {}
}

class SetPriorityFilterEvent {
  constructor(public readonly priority: 'all' | 'low' | 'medium' | 'high') {}
}

type TodoEvent =
  | AddTodoEvent
  | ToggleTodoEvent
  | SetFilterEvent
  | SetPriorityFilterEvent;

class TodoBloc extends Bloc<TodoState, TodoEvent> {
  constructor() {
    super({
      todos: [
        { id: 1, text: 'Review pull requests', completed: false, priority: 'high' },
        { id: 2, text: 'Update docs', completed: false, priority: 'medium' },
      ],
      filter: 'all',
      priorityFilter: 'all',
      nextId: 3,
    });

    // Register event handlers
    this.on(AddTodoEvent, this.handleAdd);
    this.on(ToggleTodoEvent, this.handleToggle);
    this.on(SetFilterEvent, this.handleSetFilter);
    this.on(SetPriorityFilterEvent, this.handleSetPriorityFilter);
  }

  private handleAdd = (event: AddTodoEvent, emit: (state: TodoState) => void) => {
    if (!event.text.trim()) return;

    emit({
      ...this.state,
      todos: [
        ...this.state.todos,
        {
          id: this.state.nextId,
          text: event.text.trim(),
          completed: false,
          priority: event.priority,
        },
      ],
      nextId: this.state.nextId + 1,
    });
  };

  private handleToggle = (event: ToggleTodoEvent, emit: (state: TodoState) => void) => {
    emit({
      ...this.state,
      todos: this.state.todos.map((todo) =>
        todo.id === event.id ? { ...todo, completed: !todo.completed } : todo
      ),
    });
  };

  private handleSetFilter = (event: SetFilterEvent, emit: (state: TodoState) => void) => {
    emit({
      ...this.state,
      filter: event.filter,
    });
  };

  private handleSetPriorityFilter = (
    event: SetPriorityFilterEvent,
    emit: (state: TodoState) => void
  ) => {
    emit({
      ...this.state,
      priorityFilter: event.priority,
    });
  };

  // Public API methods
  addTodo = (text: string, priority: 'low' | 'medium' | 'high' = 'medium') => {
    this.add(new AddTodoEvent(text, priority));
  };

  toggleTodo = (id: number) => {
    this.add(new ToggleTodoEvent(id));
  };

  setFilter = (filter: 'all' | 'active' | 'completed') => {
    this.add(new SetFilterEvent(filter));
  };

  setPriorityFilter = (priority: 'all' | 'low' | 'medium' | 'high') => {
    this.add(new SetPriorityFilterEvent(priority));
  };

  // ✅ Computed properties (not stored in state)
  get filteredTodos(): Todo[] {
    let filtered = this.state.todos;

    // Apply completion filter
    switch (this.state.filter) {
      case 'active':
        filtered = filtered.filter((t) => !t.completed);
        break;
      case 'completed':
        filtered = filtered.filter((t) => t.completed);
        break;
    }

    // Apply priority filter
    if (this.state.priorityFilter !== 'all') {
      filtered = filtered.filter((t) => t.priority === this.state.priorityFilter);
    }

    return filtered;
  }

  get activeCount(): number {
    return this.state.todos.filter((t) => !t.completed).length;
  }

  get completedCount(): number {
    return this.state.todos.filter((t) => t.completed).length;
  }
}
```

<GuideArticleNote title="Use Computed Properties, Not Derived State">
**Don't store filtered lists in state**:

```typescript
// ❌ WRONG: Duplicated data
interface State {
  todos: Todo[];
  filteredTodos: Todo[]; // Duplication! Must be kept in sync
}

// ✅ CORRECT: Compute on-demand
class TodoBloc extends Bloc<State> {
  get filteredTodos(): Todo[] {
    return this.state.todos.filter(/* filtering logic */);
  }
}
```

Computed properties are calculated on-demand, preventing data duplication and keeping your state minimal.
</GuideArticleNote>

## Pattern 3: Bulk Operations

When users need to act on multiple items simultaneously, implement **bulk operations**. Common in:
- Admin panels (bulk delete users)
- Email clients (select all, archive selected)
- File managers (select multiple, move to folder)
- Shopping carts (remove multiple items)

```typescript
interface Item {
  id: number;
  name: string;
  selected: boolean; // Track selection state
}

interface BulkState {
  items: Item[];
  nextId: number;
}

class BulkOperationsCubit extends Cubit<BulkState> {
  constructor() {
    super({
      items: [
        { id: 1, name: 'Item 1', selected: false },
        { id: 2, name: 'Item 2', selected: false },
        { id: 3, name: 'Item 3', selected: false },
      ],
      nextId: 4,
    });
  }

  toggleItem = (id: number) => {
    this.patch({
      items: this.state.items.map((item) =>
        item.id === id ? { ...item, selected: !item.selected } : item
      ),
    });
  };

  // Bulk operations
  selectAll = () => {
    this.patch({
      items: this.state.items.map((item) => ({ ...item, selected: true })),
    });
  };

  deselectAll = () => {
    this.patch({
      items: this.state.items.map((item) => ({ ...item, selected: false })),
    });
  };

  deleteSelected = () => {
    this.patch({
      items: this.state.items.filter((item) => !item.selected),
    });
  };

  // Computed properties
  get selectedCount(): number {
    return this.state.items.filter((item) => item.selected).length;
  }

  get allSelected(): boolean {
    return this.state.items.length > 0 && this.state.items.every((item) => item.selected);
  }

  get noneSelected(): boolean {
    return this.state.items.every((item) => !item.selected);
  }
}
```

## Best Practices

### 1. Generate Stable, Unique IDs

**Never use array indices as keys**:

```typescript
// ❌ WRONG: Index as key breaks when reordering
{items.map((item, index) => (
  <div key={index}>{item.name}</div>
))}

// ✅ CORRECT: Stable unique ID
{items.map((item) => (
  <div key={item.id}>{item.name}</div>
))}
```

**Options for generating IDs:**

```typescript
// Option 1: Auto-incrementing (simple, works for client-only)
class ListCubit extends Cubit<State> {
  private nextId = 1;

  addItem = (text: string) => {
    this.patch({
      items: [...this.state.items, { id: this.nextId++, text }],
    });
  };
}

// Option 2: UUIDs (better for distributed systems)
import { nanoid } from 'nanoid';

addItem = (text: string) => {
  this.patch({
    items: [...this.state.items, { id: nanoid(), text }],
  });
};

// Option 3: Server-generated (best for persisted data)
addItem = async (text: string) => {
  const tempId = `temp-${Date.now()}`;
  this.patch({
    items: [...this.state.items, { id: tempId, text, pending: true }],
  });

  const response = await api.createItem(text);
  this.patch({
    items: this.state.items.map(item =>
      item.id === tempId ? { ...item, id: response.id, pending: false } : item
    ),
  });
};
```

### 2. Normalize Nested Data

**Avoid deeply nested structures**:

```typescript
// ❌ BAD: Nested duplicated data
interface State {
  users: Array<{
    id: number;
    name: string;
    posts: Array<{ id: number; title: string }>; // Duplication!
  }>;
}

// ✅ GOOD: Normalized with references
interface State {
  users: Record<number, User>;
  posts: Record<number, Post>;
  userPostIds: Record<number, number[]>; // User ID → Post IDs
}

// Updating is now simple and efficient
updatePost = (postId: number, title: string) => {
  this.patch({
    posts: {
      ...this.state.posts,
      [postId]: { ...this.state.posts[postId], title },
    },
  });
};
```

### 3. Batch Multiple Updates

**Don't trigger re-renders for every item**:

```typescript
// ❌ BAD: Multiple state updates
markAllComplete = () => {
  this.state.todos.forEach((todo) => {
    this.toggleTodo(todo.id); // Re-renders for EACH todo!
  });
};

// ✅ GOOD: Single batched update
markAllComplete = () => {
  this.patch({
    todos: this.state.todos.map((todo) => ({ ...todo, completed: true })),
  });
};
```

<GuideArticleNote title="Performance with Large Lists">
For lists with **10,000+ items**, consider these optimizations:

1. **Virtual Scrolling**: Only render visible items (use `react-window` or `react-virtualized`)
2. **Pagination**: Load data in chunks
3. **Web Workers**: Move heavy operations off the main thread
4. **Memoization**: Cache expensive computed properties
5. **Debounce filters**: Delay filtering until user stops typing

```typescript
// Example: Debounced search filter
private searchTimeout: NodeJS.Timeout | null = null;

setSearchFilter = (query: string) => {
  if (this.searchTimeout) clearTimeout(this.searchTimeout);

  this.searchTimeout = setTimeout(() => {
    this.patch({ searchQuery: query });
  }, 300); // Wait 300ms after user stops typing
};
```
</GuideArticleNote>

## Testing List Operations

Here's how to test list management logic:

```typescript
import { describe, it, expect, beforeEach } from 'vitest';

describe('SimpleListCubit', () => {
  let cubit: SimpleListCubit;

  beforeEach(() => {
    cubit = new SimpleListCubit();
  });

  it('should add a new task', () => {
    const initialLength = cubit.state.tasks.length;
    cubit.addTask('New task');

    expect(cubit.state.tasks).toHaveLength(initialLength + 1);
    expect(cubit.state.tasks[cubit.state.tasks.length - 1].text).toBe('New task');
    expect(cubit.state.tasks[cubit.state.tasks.length - 1].completed).toBe(false);
  });

  it('should not add empty tasks', () => {
    const initialLength = cubit.state.tasks.length;
    cubit.addTask('   '); // Whitespace only

    expect(cubit.state.tasks).toHaveLength(initialLength);
  });

  it('should toggle task completion', () => {
    const taskId = cubit.state.tasks[0].id;
    const initialCompleted = cubit.state.tasks[0].completed;

    cubit.toggleTask(taskId);

    const updatedTask = cubit.state.tasks.find(t => t.id === taskId);
    expect(updatedTask?.completed).toBe(!initialCompleted);
  });

  it('should remove a task', () => {
    const taskId = cubit.state.tasks[0].id;
    cubit.removeTask(taskId);

    expect(cubit.state.tasks.find(t => t.id === taskId)).toBeUndefined();
  });

  it('should clear all completed tasks', () => {
    cubit.addTask('Task to complete');
    const newTaskId = cubit.state.nextId - 1;
    cubit.toggleTask(newTaskId);

    cubit.clearCompleted();

    expect(cubit.state.tasks.every(t => !t.completed)).toBe(true);
  });

  it('should generate unique IDs', () => {
    cubit.addTask('Task 1');
    cubit.addTask('Task 2');

    const ids = cubit.state.tasks.map(t => t.id);
    const uniqueIds = new Set(ids);

    expect(uniqueIds.size).toBe(ids.length); // All IDs are unique
  });
});
```

## Key Takeaways

- **Immutability is non-negotiable**: Always create new arrays/objects, never mutate directly
- **Use Cubit** for simple CRUD operations without complex logic
- **Use Bloc** for event-driven lists with filtering, audit logs, or complex business rules
- **Computed properties** prevent state duplication—calculate derived data on-demand
- **Generate stable IDs** using auto-increment, UUIDs, or server-generated IDs
- **Normalize nested data** to avoid duplication and simplify updates
- **Batch multiple updates** into a single `patch()` call for optimal performance
- **Test edge cases**: empty inputs, duplicate IDs, bulk operations on empty lists
