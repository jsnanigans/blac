## Introduction

One of the most powerful features of BlaC is **shared state**. Multiple components can read from the same Cubit, and they all stay perfectly synchronized—no prop drilling, no context boilerplate, no complex setup.

In this guide, you'll see how three completely separate components can all display the same message. When the message changes, all three update instantly and automatically.

### What You'll Learn

- How multiple components read from the same Cubit
- Why BlaC shares instances by default
- How `useBloc` automatically subscribes components
- The power of shared state without prop drilling

## The Power of Shared State

Imagine building a dashboard where multiple widgets need to display user information. In traditional React, you'd either:

1. **Prop drill** through multiple layers (messy and fragile)
2. **Use Context** (requires Provider setup and causes unnecessary re-renders)
3. **Lift state up** to a common parent (breaks component independence)

With BlaC, you simply use the same Cubit in any component that needs the data. It's that simple.

## The Message Cubit

Let's start with a Cubit that holds a simple message:

```typescript
interface MessageState {
  text: string;
  timestamp: number;
}

class MessageCubit extends Cubit<MessageState> {
  constructor() {
    super({
      text: 'Welcome to BlaC!',
      timestamp: Date.now(),
    });
  }

  updateMessage = (newText: string) => {
    this.emit({
      text: newText,
      timestamp: Date.now(),
    });
  };
}
```

This Cubit manages two pieces of data: the message text and when it was last updated.

## Multiple Components, Same State

Now let's create three components that all read from `MessageCubit`:

```tsx
function HeaderDisplay() {
  const [state] = useBloc(MessageCubit);

  return (
    <header>
      <h1>{state.text}</h1>
    </header>
  );
}

function SidebarDisplay() {
  const [state] = useBloc(MessageCubit);

  return (
    <aside>
      <p>Sidebar: {state.text}</p>
    </aside>
  );
}

function FooterDisplay() {
  const [state] = useBloc(MessageCubit);

  return (
    <footer>
      <small>{state.text}</small>
    </footer>
  );
}
```

Notice something remarkable: **each component independently calls `useBloc(MessageCubit)`**, yet they all get the same state. No providers, no props, no configuration needed.

<GuideArticleNote title="Shared by Default">
BlaC creates **one instance** of each Cubit class and shares it across all components by default. This is the most common use case and makes state management incredibly simple.

You can override this behavior with `static isolated = true` if you need separate instances (covered in the Instance Management guide).
</GuideArticleNote>

## How It Works: The useBloc Hook

When you call `useBloc(MessageCubit)` in a component, BlaC performs these steps:

1. **Find or Create Instance**: BlaC maintains a global registry. If `MessageCubit` doesn't exist yet, it creates one. If it does exist, it returns the existing instance.

2. **Subscribe Component**: Your component is added to the Cubit's subscriber list. When state changes, your component will be notified.

3. **Return Current State**: The hook returns the current state immediately, so your component can render.

4. **Automatic Re-renders**: When `updateMessage()` is called and state changes, BlaC notifies all subscribed components, triggering React re-renders.

This all happens automatically—you just call the hook and everything works.

### What's Happening

All three components are reading from the **same MessageCubit instance**. When you click a button:

1. The button calls `cubit.updateMessage(newText)`
2. MessageCubit emits the new state
3. BlaC notifies **all** subscribed components (Header, Sidebar, Footer)
4. React re-renders each component with the new state
5. All three components show the updated message

<GuideArticleNote title="Performance Note">
BlaC only re-renders components that are actually subscribed to the Cubit. Components that don't use `useBloc(MessageCubit)` won't re-render, even if they're siblings or parents of components that do.

This is thanks to BlaC's automatic dependency tracking—you get optimal performance without manual optimization.
</GuideArticleNote>

## Reading Specific Parts of State

BlaC automatically tracks which **root-level properties** you access, but there's an important limitation to understand:

<GuideArticleNote title="Dependency Tracking Limitation">
**Critical**: BlaC's dependency tracking only works on the **first layer (root level)** of state objects.

```tsx
// ✅ Good: Flat state - tracking works perfectly
interface UserState {
  userName: string;
  userEmail: string;
  themeSetting: string;
  unreadCount: number;
}

function UserName() {
  const [state] = useBloc(UserCubit);
  // Only re-renders when userName changes!
  return <h1>{state.userName}</h1>;
}

// ❌ Problem: Nested state - tracking only works at root level
interface UserState {
  profile: { name: string; email: string };
  preferences: { theme: string };
  notifications: { unread: number };
}

function UserName() {
  const [state] = useBloc(UserCubit);
  // Re-renders when ANY part of state.profile changes!
  // BlaC only tracks state.profile, not state.profile.name
  return <h1>{state.profile.name}</h1>;
}
```

**Recommendation**: Use flat state structures whenever possible:
- Flat state enables fine-grained re-render optimization
- Flat state makes updates simpler (no nested spreading)
- Flat state is easier to reason about
</GuideArticleNote>

With flat state, you get **true fine-grained reactivity** without selectors or manual optimization. Components only re-render when their specific root-level properties change.

## No Prop Drilling

Traditional React requires passing props through intermediate components:

```tsx
// ❌ Traditional prop drilling
function App() {
  const [message, setMessage] = useState('Hello');

  return <Header message={message} />;
}

function Header({ message }: { message: string }) {
  return <Nav message={message} />;
}

function Nav({ message }: { message: string }) {
  return <Title message={message} />;
}

function Title({ message }: { message: string }) {
  return <h1>{message}</h1>;
}
```

With BlaC, components access state directly:

```tsx
// ✅ No prop drilling with BlaC
function App() {
  return <Header />;
}

function Header() {
  return <Nav />;
}

function Nav() {
  return <Title />;
}

function Title() {
  const [state] = useBloc(MessageCubit);
  return <h1>{state.text}</h1>;
}
```

The intermediate components (`Header`, `Nav`) don't need to know about or pass the message. Only `Title` needs it, so only `Title` accesses it.

## Common Patterns

### Pattern 1: Display-Only Components

Many components only need to read state, not modify it:

```tsx
function DisplayMessage() {
  const [state] = useBloc(MessageCubit);
  // We don't need the cubit instance, just the state
  return <div>{state.text}</div>;
}
```

### Pattern 2: Controller Components

Some components only modify state, they don't display it:

```tsx
function MessageControls() {
  const [, cubit] = useBloc(MessageCubit);
  // We don't need state, just the cubit instance

  return (
    <button onClick={() => cubit.updateMessage('New message')}>
      Update
    </button>
  );
}
```

Notice the destructuring: `[, cubit]` - we skip the state and only take the cubit instance.

### Pattern 3: Dashboard Layout

Multiple widgets reading different parts of state:

```tsx
function Dashboard() {
  return (
    <div>
      <UserProfile />
      <NotificationCenter />
      <ActivityFeed />
      <Settings />
    </div>
  );
}

// Each component independently accesses what it needs
// Note: Using flat state for optimal dependency tracking!
function UserProfile() {
  const [state] = useBloc(UserCubit);
  return <div>{state.userName}</div>; // Flat: state.userName
}

function NotificationCenter() {
  const [state] = useBloc(NotificationsCubit);
  return <div>{state.unreadCount} new</div>; // Flat: state.unreadCount
}

function ActivityFeed() {
  const [state] = useBloc(ActivityCubit);
  return <ul>{state.items.map(/* ... */)}</ul>; // Arrays are fine
}

function Settings() {
  const [state] = useBloc(SettingsCubit);
  return <div>{state.theme}</div>; // Flat: state.theme
}
```

Each component uses a different Cubit, and they all work independently. No central store, no action dispatchers, no reducers—just clean, direct state access.

## Key Takeaways

- **Shared by default**: One Cubit instance is automatically shared across all components
- **useBloc subscribes**: Calling `useBloc` subscribes your component to state changes
- **Automatic updates**: When state changes, all subscribed components re-render automatically
- **No prop drilling**: Components access state directly, no matter where they are in the tree
- **Root-level tracking**: BlaC tracks which root-level properties you use—prefer flat state for fine-grained reactivity
- **Clean separation**: State logic lives in Cubits, UI logic in components
