## Introduction

Real applications aren't built with single components—they're composed of many components working together. Some need to share state, some need to communicate, and some need to be completely independent.

In this guide, you'll learn how to organize multiple components that work together using BlaC's shared state management.

### What You'll Learn

- How to structure parent-child component relationships
- When components should share state vs. have their own
- Component communication patterns
- Building complex UIs with simple, composable pieces
- Avoiding unnecessary coupling

## Shared State Across Components

The simplest way to coordinate multiple components is to have them all use the same Cubit. You've already seen this, `useBloc(MyCubit)` gives every component access to the same shared instance.

```tsx
interface ShoppingCartState {
  items: { id: string; name: string; price: number }[];
  discount: number;
}

class ShoppingCartCubit extends Cubit<ShoppingCartState> {
  constructor() {
    super({ items: [], discount: 0 });
  }

  addItem = (item: { id: string; name: string; price: number }) => {
    const items = [...this.state.items, item];
    this.patch({ items });
  };

  removeItem = (id: string) => {
    const items = this.state.items.filter(i => i.id !== id);
    this.patch({ items });
  };
}
```

Now multiple components can work with this shared cart:

```tsx
// Shows the cart badge with item count
function CartBadge() {
  const [state] = useBloc(ShoppingCartCubit);

  return (
    <div>
      {state.items.length}
    </div>
  );
}

// Shows the cart total
function CartTotal() {
  const [state] = useBloc(ShoppingCartCubit);
  const total = state.items.reduce((sum, i) => sum + i.price, 0);

  return (
    <div>
      ${total.toFixed(2)}
    </div>
  );
}

// Lists all items in the cart
function CartItems() {
  const [state, cart] = useBloc(ShoppingCartCubit);

  return (
    <ul>
      {state.items.map(item => (
        <li key={item.id}>
          {item.name} - ${item.price}
          <button onClick={() => cart.removeItem(item.id)}>
            Remove
          </button>
        </li>
      ))}
    </ul>
  );
}

// Product button that adds to cart
function ProductCard({ product }: { product: Product }) {
  const [, cart] = useBloc(ShoppingCartCubit);

  return (
    <div>
      <h3>{product.name}</h3>
      <p>${product.price}</p>
      <button onClick={() => cart.addItem(product)}>
        Add to Cart
      </button>
    </div>
  );
}
```

All four components work together seamlessly. When `ProductCard` adds an item, `CartBadge`, `CartTotal`, and `CartItems` all update automatically.

<GuideArticleNote title="Component Independence">
Notice how each component is **completely independent**. They don't know about each other. They don't pass props. They don't call each other's methods. There is no Provider required.

They only share a Cubit. This makes components:
- **Easy to move** - Drop them anywhere in your tree
- **Easy to test** - Test each component in isolation
- **Easy to understand** - Each component has one clear responsibility
</GuideArticleNote>

## Parent-Child Communication

Sometimes you need parent components to coordinate child components. There are several patterns for this:

### Pattern: Siblings Coordinate Through Shared State

Components at the same level coordinate by reading/writing shared state:

```tsx
interface FilterState {
  searchTerm: string;
  category: string | null;
  priceRange: [number, number];
}

class FilterCubit extends Cubit<FilterState> {
  constructor() {
    super({
      searchTerm: '',
      category: null,
      priceRange: [0, 1000],
    });
  }

  setSearch = (searchTerm: string) => {
    this.patch({ searchTerm });
  };

  setCategory = (category: string | null) => {
    this.patch({ category });
  };

  setPriceRange = (priceRange: [number, number]) => {
    this.patch({ priceRange });
  };

  reset = () => {
    this.emit({
      searchTerm: '',
      category: null,
      priceRange: [0, 1000],
    });
  };

  // Filter products based on filter state
  get filteredProducts() {
    return this.state.products.filter(product => {
      const matchesSearch = product.name
        .toLowerCase()
        .includes(filters.searchTerm.toLowerCase());
      const matchesCategory = !filters.category ||
        product.category === filters.category;
      const matchesPrice = product.price >= filters.priceRange[0] &&
        product.price <= filters.priceRange[1];

      return matchesSearch && matchesCategory && matchesPrice;
    });
  }
}

// Sibling components that coordinate
function FilterBar() {
  return (
    <div>
      <SearchBox />
      <CategoryDropdown />
      <PriceSlider />
      <ResetButton />
    </div>
  );
}

function SearchBox() {
  const [state, filter] = useBloc(FilterCubit);

  return (
    <input
      value={state.searchTerm}
      onChange={(e) => filter.setSearch(e.target.value)}
      placeholder="Search products..."
    />
  );
}

function CategoryDropdown() {
  const [state, filter] = useBloc(FilterCubit);

  return (
    <select
      value={state.category ?? ''}
      onChange={(e) => filter.setCategory(e.target.value || null)}
    >
      <option value="">All Categories</option>
      <option value="electronics">Electronics</option>
      <option value="books">Books</option>
      <option value="clothing">Clothing</option>
    </select>
  );
}

function ProductList() {
  const [, { filteredProducts }] = useBloc(FilterCubit);

  return (
    <div>
      {filteredProducts.map(product => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  );
}
```

`FilterBar` and `ProductList` are siblings. They don't know about each other, but they coordinate through `FilterCubit`.

<GuideArticleNote title="When to Share State">
Components should share a Cubit when they need to:
- **Display the same data** (cart badge and cart total)
- **Coordinate behavior** (filters and product list)
- **Maintain consistency** (form inputs and submit button)

If components don't need to coordinate, give them separate Cubits or make them `isolated` (covered in Instance Management).
</GuideArticleNote>

## Composition Patterns

### Pattern: Layout Components with Smart Children

Layout components don't need state—they just arrange children:

```tsx
// Layout component (no state)
function DashboardLayout() {
  return (
    <div>
      <Sidebar />
      <MainContent />
      <Notifications />
    </div>
  );
}

// Each child manages its own state
function Sidebar() {
  const [state] = useBloc(NavigationCubit);
  return <nav>{/* navigation */}</nav>;
}

function MainContent() {
  const [state] = useBloc(ContentCubit);
  return <main>{/* content */}</main>;
}

function Notifications() {
  const [state] = useBloc(NotificationsCubit);
  return <aside>{/* notifications */}</aside>;
}
```

The layout doesn't care what the children do. Each child has its own Cubit.

### Pattern: Conditional Rendering Based on State

Components can mount/unmount based on shared state:

```tsx
interface AppState {
  isAuthenticated: boolean;
  user: User | null;
}

class AppCubit extends Cubit<AppState> {
  constructor() {
    super({ isAuthenticated: false, user: null });
  }

  login = (user: User) => {
    this.patch({ isAuthenticated: true, user });
  };

  logout = () => {
    this.emit({ isAuthenticated: false, user: null });
  };
}

function App() {
  const [state] = useBloc(AppCubit);

  return (
    <div>
      <Header />
      {state.isAuthenticated ? (
        <Dashboard />
      ) : (
        <LoginForm />
      )}
    </div>
  );
}

function Header() {
  const [state, app] = useBloc(AppCubit);

  return (
    <header>
      <h1>My App</h1>
      {state.isAuthenticated && (
        <button onClick={app.logout}>Logout</button>
      )}
    </header>
  );
}
```

`App` and `Header` both use `AppCubit` to coordinate authentication state.

## Avoiding Prop Drilling

One of BlaC's biggest advantages is eliminating prop drilling. Compare these approaches:

### ❌ Traditional Prop Drilling

```tsx
function App() {
  const [theme, setTheme] = useState('light');

  return <Layout theme={theme} onThemeChange={setTheme} />;
}

function Layout({ theme, onThemeChange }: Props) {
  return (
    <div>
      <Sidebar theme={theme} onThemeChange={onThemeChange} />
      <Content theme={theme} />
    </div>
  );
}

function Sidebar({ theme, onThemeChange }: Props) {
  return (
    <aside>
      <Nav theme={theme} />
      <ThemeToggle theme={theme} onThemeChange={onThemeChange} />
    </aside>
  );
}

function ThemeToggle({ theme, onThemeChange }: Props) {
  return (
    <button onClick={() => onThemeChange(theme === 'light' ? 'dark' : 'light')}>
      Toggle Theme
    </button>
  );
}
```

Every intermediate component needs to accept and pass props, even if they don't use them.

### ✅ BlaC: Direct Access

```tsx
class ThemeCubit extends Cubit<{ theme: 'light' | 'dark' }> {
  constructor() {
    super({ theme: 'light' });
  }

  toggleTheme = () => {
    this.patch({
      theme: this.state.theme === 'light' ? 'dark' : 'light',
    });
  };
}

function App() {
  return <Layout />;
}

function Layout() {
  return (
    <div>
      <Sidebar />
      <Content />
    </div>
  );
}

function Sidebar() {
  return (
    <aside>
      <Nav />
      <ThemeToggle />
    </aside>
  );
}

function ThemeToggle() {
  const [, { toggleTheme }] = useBloc(ThemeCubit);

  return (
    <button onClick={toggleTheme}>
      Toggle Theme
    </button>
  );
}

function Nav() {
  const [{ theme }] = useBloc(ThemeCubit);

  return <nav className={theme}>{/* nav items */}</nav>;
}
```

Only components that need the theme access it. No props drilling through intermediate components.

## Key Takeaways

- **Components share Cubits naturally** - Just use `useBloc` wherever you need the state
- **No prop drilling required** - Components access state directly
- **Components stay independent** - They don't need to know about each other
- **Composition is easy** - Build complex UIs from simple, focused components
- **Each component has one job** - Display state, update state, or orchestrate children
- **Testing is easier** - Simple unit tests for your logic code in the Cubit
