## Introduction

You've learned how to read state—now it's time to master **updating** it. BlaC gives you two methods: `emit()` and `patch()`. They look similar but behave very differently.

Understanding when to use each one is crucial. Use the wrong method, and you might accidentally lose data or write verbose, repetitive code.

### What You'll Learn

- The difference between `emit()` and `patch()`
- When to use each method
- How shallow merging works
- Common mistakes with nested objects
- Performance implications

## Two Methods, Two Purposes

Think of state updates like replacing a document vs. making edits:

- **emit()** = "Replace the entire document"
- **patch()** = "Edit specific paragraphs, keep the rest"

Both trigger React re-renders. Both update your state. But they work fundamentally differently.

## emit(): Complete Replacement

When you call `emit()`, you replace the **entire state object** with a brand new one. Every field must be provided, even if it hasn't changed.

```typescript
interface ProfileState {
  name: string;
  age: number;
  email: string;
}

class ProfileCubit extends Cubit<ProfileState> {
  constructor() {
    super({
      name: 'Alice',
      age: 28,
      email: 'alice@example.com',
    });
  }

  loadNewProfile = () => {
    this.emit({
      name: 'Bob',         // Must provide all fields
      age: 35,
      email: 'bob@example.com',
    });
  };

  reset = () => {
    this.emit({
      name: 'Alice',
      age: 28,
      email: 'alice@example.com',
    });
  };
}
```

Notice how both methods provide **every field**, not just the ones that changed.

<GuideArticleNote title="When to Use emit()">
Use `emit()` when you're:
- **Loading fresh data** from an API (you receive a complete new object)
- **Resetting to defaults** (returning to initial state)
- **Replacing everything** (transforming the entire state)

It's like getting a new piece of paper—you write everything from scratch.
</GuideArticleNote>

## patch(): Partial Updates

When you call `patch()`, BlaC performs a **shallow merge**. You only provide the fields you want to change. Everything else stays the same.

```typescript
class ProfileCubit extends Cubit<ProfileState> {
  updateName = (name: string) => {
    this.patch({ name });
    // age and email stay the same!
  };

  updateAge = (age: number) => {
    this.patch({ age });
    // name and email stay the same!
  };

  updateMultiple = () => {
    this.patch({
      name: 'Charlie',
      age: 30,
    });
    // email stays the same!
  };
}
```

This is much more convenient for individual field updates—you don't have to repeat unchanged values.

<GuideArticleNote title="When to Use patch()">
Use `patch()` when you're:
- **Updating form inputs** (change one field at a time)
- **Toggling flags** (flip a single boolean)
- **Incrementing counters** (modify just one number)
- **Changing individual properties** (without affecting others)

It's like using an eraser and pencil—you change specific parts, not the whole thing.
</GuideArticleNote>

## The Shallow Merge Trap

Here's the most common mistake with `patch()`: **it only merges at the top level**. Nested objects are replaced entirely, not merged.

### ❌ Wrong: Loses Nested Data

```typescript
interface UserState {
  name: string;
  preferences: {
    theme: 'light' | 'dark';
    notifications: boolean;
    language: string;
  };
}

class UserCubit extends Cubit<UserState> {
  toggleTheme = () => {
    // ❌ This LOSES notifications and language!
    this.patch({
      preferences: {
        theme: this.state.preferences.theme === 'light' ? 'dark' : 'light',
      }
    });
    // Result: { name: "Alice", preferences: { theme: "dark" } }
    // notifications and language are gone!
  };
}
```

The `preferences` object is **completely replaced**, not merged. Only `theme` survives.

### ✅ Correct: Spread to Preserve

```typescript
class UserCubit extends Cubit<UserState> {
  toggleTheme = () => {
    // ✅ Spread to keep other preferences
    this.patch({
      preferences: {
        ...this.state.preferences,
        theme: this.state.preferences.theme === 'light' ? 'dark' : 'light',
      }
    });
    // Result: All preferences preserved, only theme changes
  };
}
```

The spread operator `...this.state.preferences` copies all existing fields before adding the new `theme` value.

<GuideArticleNote title="Critical: Always Spread Nested Objects">
When using `patch()` with nested objects, **always use the spread operator** to preserve other fields:

```typescript
// ❌ Wrong - loses data
this.patch({ nested: { field: 'new' } });

// ✅ Correct - preserves data
this.patch({ nested: { ...this.state.nested, field: 'new' } });
```

This is the #1 source of bugs with `patch()`. Don't skip the spread!
</GuideArticleNote>

<GuideArticleNote title="Nested State Has Two Problems">
Nested state creates **two challenges**:

1. **Update Complexity**: You must spread nested objects to avoid data loss (as shown above)
2. **Dependency Tracking**: BlaC only tracks **root-level properties**, not nested ones

```typescript
// ❌ Problem: Changes to preferences.theme trigger re-renders in ALL components
// that access state.preferences, even if they only use state.preferences.language
function ThemeDisplay() {
  const [state] = useBloc(UserCubit);
  return <div>{state.preferences.theme}</div>; // Tracks state.preferences
}
```

**Recommendation**: Use flat state structures when possible:
```typescript
// ✅ Better: Flat state enables fine-grained tracking
interface UserState {
  name: string;
  preferencesTheme: 'light' | 'dark';
  preferencesNotifications: boolean;
  preferencesLanguage: string;
}
// Now components only re-render when their specific fields change!
```

Flat state is easier to update AND enables optimal dependency tracking!
</GuideArticleNote>

## Comparison: Side by Side

| Feature | emit() | patch() |
|---------|--------|---------|
| **What it does** | Replaces entire state | Merges specific fields |
| **Fields required** | All fields | Only changed fields |
| **Nested objects** | Handles naturally | Requires spreading |
| **Best for** | Resets, API loads | Form inputs, toggles |
| **Verbosity** | High for small changes | Low for small changes |
| **Risk** | None (explicit) | Data loss if not careful |

## Real-World Examples

### Example 1: Form Handling

```typescript
interface FormState {
  username: string;
  email: string;
  bio: string;
  agreeToTerms: boolean;
}

class FormCubit extends Cubit<FormState> {
  // ✅ Perfect use of patch() - individual field updates
  updateUsername = (username: string) => {
    this.patch({ username });
  };

  updateEmail = (email: string) => {
    this.patch({ email });
  };

  toggleTerms = () => {
    this.patch({ agreeToTerms: !this.state.agreeToTerms });
  };

  // ✅ emit() for complete reset
  resetForm = () => {
    this.emit({
      username: '',
      email: '',
      bio: '',
      agreeToTerms: false,
    });
  };
}
```

### Example 2: API Data Loading

```typescript
interface ProductState {
  id: string;
  name: string;
  price: number;
  stock: number;
}

class ProductCubit extends Cubit<ProductState | null> {
  // ✅ emit() perfect for loading complete API response
  loadProduct = async (id: string) => {
    const product = await fetchProduct(id);
    this.emit(product); // Replace everything
  };

  // ✅ patch() for individual updates
  updateStock = (stock: number) => {
    if (this.state) {
      this.patch({ stock });
    }
  };
}
```

### Example 3: Counter (Simple State)

```typescript
// When state is just a number, emit() and patch() are similar
class CounterCubit extends Cubit<number> {
  increment = () => {
    this.emit(this.state + 1); // Common for primitives
  };
}

// But for object state, patch() is cleaner
interface CountState {
  count: number;
}

class CountObjectCubit extends Cubit<CountState> {
  increment = () => {
    this.patch({ count: this.state.count + 1 }); // Cleaner!
  };
}
```

## Performance Considerations

Both `emit()` and `patch()` trigger React re-renders. There's no performance difference between them—they're equally fast.

The real performance consideration is **how many fields changed**:

- Changing 1 field with `emit()` → React re-renders
- Changing 1 field with `patch()` → React re-renders (same performance)
- Changing 10 fields with either → React re-renders (same performance)

BlaC's dependency tracking optimizes re-renders based on **what fields components actually use**, not which update method you chose.

<GuideArticleNote title="Pro Tip: Favor patch()">
In practice, you'll use `patch()` much more often than `emit()`. Most updates only change a few fields, not the entire state.

Reserve `emit()` for resets, API loads, and complete transformations.
</GuideArticleNote>

## Deep Nesting: When to Reconsider

If you find yourself writing code like this:

```typescript
this.patch({
  user: {
    ...this.state.user,
    profile: {
      ...this.state.user.profile,
      settings: {
        ...this.state.user.profile.settings,
        theme: 'dark',
      },
    },
  },
});
```

It's time to reconsider your state structure. Deeply nested state has **two major problems**:

1. **Hard to update** - Complex spreading is error-prone
2. **Poor performance** - Dependency tracking only works at root level, so components re-render unnecessarily

Consider these solutions:

1. **Flatten your state** - Reduce nesting levels (best for dependency tracking!)
2. **Split into multiple Cubits** - Separate concerns
3. **Use helper methods** - Encapsulate complex updates (doesn't fix dependency tracking)

```typescript
// Better: Helper method encapsulates complexity
updateTheme = (theme: 'light' | 'dark') => {
  this.patch({
    user: {
      ...this.state.user,
      profile: {
        ...this.state.user.profile,
        settings: {
          ...this.state.user.profile.settings,
          theme,
        },
      },
    },
  });
};

// Component just calls: cubit.updateTheme('dark')
```

## Key Takeaways

- **emit()** replaces the entire state—perfect for resets and complete updates
- **patch()** merges at the top level—perfect for individual field changes
- **Always spread nested objects** when using patch() to avoid data loss
- **Prefer flat state** - Easier to update AND enables fine-grained dependency tracking
- **Both trigger re-renders equally**—no performance difference
- **Favor patch()** for most updates—it's more convenient
- **Deep nesting has two problems**—hard to update AND breaks dependency tracking
