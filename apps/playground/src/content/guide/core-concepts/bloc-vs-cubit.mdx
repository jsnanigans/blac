# Bloc vs Cubit: Choosing the Right Pattern

When working with BlaC, one of your first decisions is whether to use a **Cubit** or a **Bloc**. Both are powerful state management tools, but they serve different purposes and have different trade-offs.

## What You'll Learn

- The fundamental differences between Cubits and Blocs
- When to use each pattern
- How to migrate between them
- Performance and testing considerations
- Common misconceptions and pitfalls

## Quick Comparison

Here's a side-by-side comparison to help you decide:

| Feature | Cubit | Bloc |
|---------|-------|------|
| **Complexity** | Simple, direct | Event-driven, more structure |
| **State Changes** | Direct method calls | Event classes trigger handlers |
| **Boilerplate** | Minimal | More setup (events, handlers) |
| **Traceability** | Method calls only | Full event history possible |
| **Testing** | Test methods directly | Test event → state transitions |
| **Async** | Handle directly in methods | Multiple emissions per event |
| **Best For** | Simple state, forms, UI toggles | Complex workflows, APIs, business logic |

## Cubit: Simple and Direct

Cubits are the simpler of the two patterns. You call methods directly on the Cubit, and those methods emit new state.

```typescript
interface CounterState {
  count: number;
}

class CounterCubit extends Cubit<CounterState> {
  constructor() {
    super({ count: 0 });
  }

  increment = () => {
    this.patch({ count: this.state.count + 1 });
  };

  decrement = () => {
    this.patch({ count: this.state.count - 1 });
  };

  reset = () => {
    this.emit({ count: 0 });
  };
}
```

**Using in React:**

```tsx
function Counter() {
  const [state, cubit] = useBloc(CounterCubit);

  return (
    <div>
      <p>Count: {state.count}</p>
      <button onClick={cubit.increment}>+</button>
      <button onClick={cubit.decrement}>-</button>
      <button onClick={cubit.reset}>Reset</button>
    </div>
  );
}
```

### When to Use Cubits

- **Simple state management**: Counters, toggles, UI state
- **Form handling**: Input values, validation state
- **Minimal boilerplate**: When you want to get started quickly
- **Direct control**: When you don't need event traceability
- **Synchronous operations**: When most operations are immediate

## Bloc: Event-Driven Architecture

Blocs introduce an event-driven layer. Instead of calling methods directly, you dispatch **event objects** that trigger **event handlers** to produce new state.

```typescript
// Define event classes
class IncrementEvent {
  constructor(public readonly amount: number = 1) {}
}

class DecrementEvent {
  constructor(public readonly amount: number = 1) {}
}

class ResetEvent {}

// Define the Bloc
class CounterBloc extends Bloc<CounterState, IncrementEvent | DecrementEvent | ResetEvent> {
  constructor() {
    super({ count: 0 });

    // Register event handlers
    this.on(IncrementEvent, (event, emit) => {
      emit({ count: this.state.count + event.amount });
    });

    this.on(DecrementEvent, (event, emit) => {
      emit({ count: this.state.count - event.amount });
    });

    this.on(ResetEvent, (event, emit) => {
      emit({ count: 0 });
    });
  }

  // Convenience methods (optional)
  increment = (amount = 1) => {
    this.add(new IncrementEvent(amount));
  };

  decrement = (amount = 1) => {
    this.add(new DecrementEvent(amount));
  };

  reset = () => {
    this.add(new ResetEvent());
  };
}
```

**Using in React:**

```tsx
function Counter() {
  const [state, bloc] = useBloc(CounterBloc);

  return (
    <div>
      <p>Count: {state.count}</p>
      {/* Can use convenience methods */}
      <button onClick={() => bloc.increment()}>+</button>
      <button onClick={() => bloc.decrement()}>-</button>
      {/* Or dispatch events directly */}
      <button onClick={() => bloc.add(new ResetEvent())}>Reset</button>
    </div>
  );
}
```

### When to Use Blocs

- **Complex business logic**: Multi-step workflows, state machines
- **API integration**: Async operations with loading/error states
- **Event traceability**: When you need to log/debug state changes
- **Event transformation**: Debouncing, throttling, cancellation
- **Multiple emissions**: When one action triggers multiple state updates
- **Team collaboration**: When explicit events improve code clarity

<GuideArticleNote title="Event Classes Enable Powerful Patterns">
Event classes aren't just for organization—they unlock advanced patterns:

- **Debouncing**: Transform search events to reduce API calls
- **Cancellation**: Cancel in-flight requests when new events arrive
- **Event History**: Log every action for debugging
- **Time Travel**: Replay events to reproduce bugs
- **Analytics**: Track user actions with structured data
</GuideArticleNote>

## Decision Matrix

Use this flowchart to choose between Cubit and Bloc:

```
Is your state management simple? (toggles, counters, form fields)
│
├─ YES → Use Cubit
│         - Less boilerplate
│         - Faster to implement
│         - Easier to understand
│
└─ NO → Is event traceability important?
        │
        ├─ YES → Use Bloc
        │         - Full event history
        │         - Better debugging
        │         - Event transformation
        │
        └─ NO → Do you need async complexity? (loading states, retries)
                │
                ├─ YES → Use Bloc
                │         - Multiple emissions per event
                │         - Clear async state transitions
                │         - Better error handling patterns
                │
                └─ NO → Use Cubit
                        - Simpler is better
                        - Can always migrate later
```

## Real-World Examples

### Use Case 1: Theme Toggle → Cubit

A simple theme switcher doesn't need event traceability:

```typescript
class ThemeCubit extends Cubit<'light' | 'dark'> {
  constructor() {
    super('light');
  }

  toggle = () => {
    this.emit(this.state === 'light' ? 'dark' : 'light');
  };
}
```

### Use Case 2: User Authentication → Bloc

Login involves multiple steps and needs clear state transitions:

```typescript
class LoginEvent {
  constructor(
    public readonly email: string,
    public readonly password: string
  ) {}
}

class LogoutEvent {}

interface AuthState {
  isAuthenticated: boolean;
  userId: string | null;
  isLoading: boolean;
  error: string | null;
}

class AuthBloc extends Bloc<AuthState, LoginEvent | LogoutEvent> {
  constructor() {
    super({
      isAuthenticated: false,
      userId: null,
      isLoading: false,
      error: null,
    });

    this.on(LoginEvent, async (event, emit) => {
      // Emit loading state
      emit({ ...this.state, isLoading: true, error: null });

      try {
        const user = await api.login(event.email, event.password);

        // Emit success state
        emit({
          isAuthenticated: true,
          userId: user.id,
          isLoading: false,
          error: null,
        });
      } catch (error) {
        // Emit error state
        emit({
          isAuthenticated: false,
          userId: null,
          isLoading: false,
          error: error.message,
        });
      }
    });

    this.on(LogoutEvent, (event, emit) => {
      emit({
        isAuthenticated: false,
        userId: null,
        isLoading: false,
        error: null,
      });
    });
  }
}
```

<GuideArticleNote title="Multiple Emissions per Event">
Notice how the `LoginEvent` handler emits state **three times** during the async flow:

1. **Loading state** when the request starts
2. **Success state** when login succeeds
3. **Error state** if login fails

This pattern is much cleaner with Blocs than with Cubits!
</GuideArticleNote>

### Use Case 3: Search Input → Start with Cubit, Migrate to Bloc

You might start with a simple Cubit:

```typescript
class SearchCubit extends Cubit<string> {
  constructor() {
    super('');
  }

  updateQuery = (query: string) => {
    this.emit(query);
  };
}
```

But as requirements grow (debouncing, analytics), migrate to a Bloc:

```typescript
class UpdateSearchEvent {
  constructor(public readonly query: string) {}
}

class SearchBloc extends Bloc<string, UpdateSearchEvent> {
  private debounceTimer: NodeJS.Timeout | null = null;

  constructor() {
    super('');

    this.on(UpdateSearchEvent, (event, emit) => {
      // Clear previous timer
      if (this.debounceTimer) {
        clearTimeout(this.debounceTimer);
      }

      // Debounce for 300ms
      this.debounceTimer = setTimeout(() => {
        emit(event.query);
        // Could also trigger analytics here
        analytics.trackSearch(event.query);
      }, 300);
    });
  }

  override dispose() {
    if (this.debounceTimer) {
      clearTimeout(this.debounceTimer);
    }
    super.dispose();
  }

  updateQuery = (query: string) => {
    this.add(new UpdateSearchEvent(query));
  };
}
```

## Migration Paths

### From Cubit to Bloc

If you start with a Cubit and need to migrate:

1. **Create event classes** for each public method
2. **Extend Bloc** instead of Cubit
3. **Move logic into event handlers** using `on()`
4. **Keep convenience methods** to avoid breaking consumers
5. **Update tests** to dispatch events

**Before (Cubit):**

```typescript
class TodoCubit extends Cubit<Todo[]> {
  constructor() {
    super([]);
  }

  addTodo = (title: string) => {
    const newTodo = { id: Date.now(), title, completed: false };
    this.emit([...this.state, newTodo]);
  };
}
```

**After (Bloc):**

```typescript
class AddTodoEvent {
  constructor(public readonly title: string) {}
}

class TodoBloc extends Bloc<Todo[], AddTodoEvent> {
  constructor() {
    super([]);

    this.on(AddTodoEvent, (event, emit) => {
      const newTodo = { id: Date.now(), title: event.title, completed: false };
      emit([...this.state, newTodo]);
    });
  }

  // Keep for backward compatibility
  addTodo = (title: string) => {
    this.add(new AddTodoEvent(title));
  };
}
```

### From Bloc to Cubit

If you realize a Bloc is overkill:

1. **Remove event classes**
2. **Extend Cubit** instead of Bloc
3. **Move handler logic into direct methods**
4. **Simplify tests** to call methods directly

This is less common but perfectly valid for simplification.

## Performance Considerations

### Memory Overhead

- **Cubits**: Minimal overhead—just the class instance and state
- **Blocs**: Additional overhead for event objects and handlers

For most apps, this difference is negligible.

### Async Performance

Blocs can be more efficient for complex async flows because:
- Multiple emissions during one async operation
- Built-in event queuing prevents race conditions
- Event transformation (debouncing) reduces unnecessary work

### Component Re-renders

Both Cubits and Blocs use the **same dependency tracking system**, so re-render performance is identical:

```tsx
// These have the same performance characteristics:
const [state, cubit] = useBloc(CounterCubit);
const [state, bloc] = useBloc(CounterBloc);
```

<GuideArticleNote title="Dependency Tracking Works the Same">
BlaC's automatic dependency tracking works identically for both Cubits and Blocs. Your choice between them **does not affect React re-render performance**.

What matters is keeping your state **flat** (root-level properties) for optimal tracking!
</GuideArticleNote>

## Testing Comparison

### Testing Cubits

Test by calling methods and asserting state:

```typescript
test('CounterCubit increments', () => {
  const cubit = new CounterCubit();

  expect(cubit.state.count).toBe(0);

  cubit.increment();
  expect(cubit.state.count).toBe(1);

  cubit.increment();
  expect(cubit.state.count).toBe(2);

  cubit.dispose();
});
```

### Testing Blocs

Test by dispatching events and asserting state transitions:

```typescript
test('CounterBloc increments with events', () => {
  const bloc = new CounterBloc();

  expect(bloc.state.count).toBe(0);

  bloc.add(new IncrementEvent(5));
  expect(bloc.state.count).toBe(5);

  bloc.add(new DecrementEvent(2));
  expect(bloc.state.count).toBe(3);

  bloc.dispose();
});
```

### Async Testing

Blocs make async testing clearer because each state transition is explicit:

```typescript
test('AuthBloc handles login flow', async () => {
  const bloc = new AuthBloc();
  const states: AuthState[] = [];

  // Collect all state emissions
  bloc.stream.listen((state) => {
    states.push(state);
  });

  bloc.add(new LoginEvent('user@example.com', 'password123'));

  // Wait for async completion
  await new Promise(resolve => setTimeout(resolve, 100));

  // Assert state transitions
  expect(states[0].isLoading).toBe(true);  // Loading state
  expect(states[1].isAuthenticated).toBe(true);  // Success state
  expect(states[1].isLoading).toBe(false);

  bloc.dispose();
});
```

## Common Misconceptions

### Misconception 1: "Blocs are always better"

**Reality**: Blocs add complexity. For simple state (toggles, counters, form fields), Cubits are often the better choice.

### Misconception 2: "Cubits can't handle async"

**Reality**: Cubits handle async just fine with async methods:

```typescript
class DataCubit extends Cubit<Data | null> {
  constructor() {
    super(null);
  }

  fetchData = async () => {
    const data = await api.fetchData();
    this.emit(data);
  };
}
```

The difference is that Blocs make **multiple state emissions** during async operations cleaner.

### Misconception 3: "You can't mix Cubits and Blocs"

**Reality**: You can (and should) use both in the same app! Use Cubits for simple state and Blocs for complex logic.

### Misconception 4: "Events make Blocs slower"

**Reality**: The overhead of event objects is negligible. The real performance bottleneck is usually in how you structure your state and components, not whether you use events.

### Misconception 5: "I need to choose one pattern for my whole app"

**Reality**: Different features have different needs. Use Cubits for UI state and Blocs for business logic. They work together seamlessly.

## Key Takeaways

- **Cubits** are simpler and perfect for straightforward state management
- **Blocs** add structure and are ideal for complex workflows and async operations
- **Both use the same dependency tracking**, so performance is equivalent
- **Start simple** (Cubit) and migrate to Bloc only when you need the extra power
- **Use both patterns** in the same app—they're designed to work together
- **Event classes** enable powerful patterns like debouncing and event history
- **Testing differs slightly**, but both are easy to test
