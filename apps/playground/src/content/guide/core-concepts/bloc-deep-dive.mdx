## Introduction

You've mastered Cubits—simple, direct state management. Now it's time to level up with **Blocs**: event-driven state containers that bring structure, traceability, and testability to complex applications.

While Cubits update state directly with methods, Blocs take a different approach: **every state change is triggered by an event**. This event-driven architecture unlocks powerful capabilities for applications with complex business logic.

### What You'll Learn

- The event-driven Bloc pattern and why it matters
- Creating and handling event classes
- Async event processing with multiple state emissions
- Event transformation patterns (debounce, throttle, cancel)
- Testing Blocs with event verification
- When to choose Blocs vs Cubits
- Flat state architecture for optimal performance

## The Event-Driven Pattern

Blocs introduce a layer of indirection between actions and state changes. Instead of calling methods that directly update state, you **dispatch events** that are processed by registered event handlers.

```typescript
// Cubit: Direct method calls
counterCubit.increment();

// Bloc: Event-driven dispatch
counterBloc.add(new IncrementEvent());
```

This indirection brings several benefits:
- **Traceability**: Every state change has a clear cause (the event)
- **Testability**: Events are easy to mock and verify
- **Debugging**: Event logs create an audit trail
- **Architecture**: Separation between UI actions and business logic

## Creating Event Classes

Events are TypeScript classes that represent actions in your application. They can carry data and are type-safe.

```typescript
// Simple event (no data)
class ResetEvent {}

// Event with data
class IncrementEvent {
  constructor(public readonly amount: number = 1) {}
}

class DecrementEvent {
  constructor(public readonly amount: number = 1) {}
}
```

Notice how events:
- Use `readonly` to prevent modification
- Can have default values (`amount = 1`)
- Are explicit about their purpose
- Carry only the data needed for that action

<GuideArticleNote title="Event Naming Convention">
Name events as **past-tense actions** or **imperative commands**:
- ✅ `IncrementEvent`, `UserLoggedInEvent`, `DataFetchedEvent`
- ❌ `Increment`, `UserLogin`, `Data`

The `Event` suffix makes code clearer and helps distinguish events from other classes.
</GuideArticleNote>

## Registering Event Handlers

In the Bloc's constructor, you register handlers for each event type using the `on()` method:

```typescript
interface CounterState {
  count: number;
  history: Array<{ event: string; timestamp: number }>;
}

class EventCounterBloc extends Bloc<CounterState, IncrementEvent | DecrementEvent | ResetEvent> {
  constructor() {
    super({ count: 0, history: [] });

    // Register handler for IncrementEvent
    this.on(IncrementEvent, (event, emit) => {
      const newHistory = [
        { event: `Increment(${event.amount})`, timestamp: Date.now() },
        ...this.state.history,
      ].slice(0, 10); // Keep last 10 events

      emit({
        count: this.state.count + event.amount,
        history: newHistory,
      });
    });

    // Register handler for DecrementEvent
    this.on(DecrementEvent, (event, emit) => {
      const newHistory = [
        { event: `Decrement(${event.amount})`, timestamp: Date.now() },
        ...this.state.history,
      ].slice(0, 10);

      emit({
        count: this.state.count - event.amount,
        history: newHistory,
      });
    });

    // Register handler for ResetEvent
    this.on(ResetEvent, (event, emit) => {
      const newHistory = [
        { event: 'Reset', timestamp: Date.now() },
        ...this.state.history,
      ].slice(0, 10);

      emit({
        count: 0,
        history: newHistory,
      });
    });
  }

  // Helper methods dispatch events
  increment = (amount = 1) => {
    this.add(new IncrementEvent(amount));
  };

  decrement = (amount = 1) => {
    this.add(new DecrementEvent(amount));
  };

  reset = () => {
    this.add(new ResetEvent());
  };
}
```

Event handlers receive:
1. **event**: The dispatched event instance with its data
2. **emit**: Function to emit new state

<GuideArticleNote title="Flat State for Performance">
Notice how this state uses **flat properties** (`count`, `history`) instead of nested objects.

BlaC's dependency tracking only works at the **root level**, so:
- ✅ `state.count` - Tracked correctly
- ❌ `state.data.count` - Only `state.data` is tracked, not `state.data.count`

**Recommendation**: Keep Bloc state flat whenever possible for optimal re-render performance!
</GuideArticleNote>

## Handling Form Events

Blocs excel at complex workflows like form validation. Here's a pattern using multiple event types:

```typescript
// Form events
class UpdateFieldEvent {
  constructor(
    public readonly field: string,
    public readonly value: string
  ) {}
}

class SubmitFormEvent {}

class ResetFormEvent {}

// Flat state for optimal dependency tracking
interface FormState {
  emailValue: string;
  emailError: string | null;
  emailTouched: boolean;
  passwordValue: string;
  passwordError: string | null;
  passwordTouched: boolean;
  isSubmitting: boolean;
  isValid: boolean;
  submitCount: number;
}

class FormValidationBloc extends Bloc<FormState, UpdateFieldEvent | SubmitFormEvent | ResetFormEvent> {
  constructor() {
    const initialState: FormState = {
      emailValue: '',
      emailError: null,
      emailTouched: false,
      passwordValue: '',
      passwordError: null,
      passwordTouched: false,
      isSubmitting: false,
      isValid: false,
      submitCount: 0,
    };

    super(initialState);

    // Update field and validate
    this.on(UpdateFieldEvent, (event, emit) => {
      let newState = { ...this.state };

      if (event.field === 'email') {
        newState.emailValue = event.value;
        newState.emailTouched = true;
        newState.emailError = this.validateEmail(event.value);
      } else if (event.field === 'password') {
        newState.passwordValue = event.value;
        newState.passwordTouched = true;
        newState.passwordError = this.validatePassword(event.value);
      }

      // Update isValid based on all fields
      newState.isValid =
        !newState.emailError &&
        !newState.passwordError &&
        newState.emailValue.length > 0 &&
        newState.passwordValue.length > 0;

      emit(newState);
    });

    // Handle async submission
    this.on(SubmitFormEvent, async (event, emit) => {
      // Emit submitting state
      emit({ ...this.state, isSubmitting: true });

      // Simulate API call
      await new Promise(resolve => setTimeout(resolve, 1500));

      // Emit success state
      emit({
        ...this.state,
        isSubmitting: false,
        submitCount: this.state.submitCount + 1,
      });
    });

    // Reset form
    this.on(ResetFormEvent, (event, emit) => {
      emit(initialState);
    });
  }

  private validateEmail(value: string): string | null {
    if (!value) return 'Email is required';
    if (!value.includes('@')) return 'Invalid email format';
    return null;
  }

  private validatePassword(value: string): string | null {
    if (!value) return 'Password is required';
    if (value.length < 8) return 'Password must be at least 8 characters';
    return null;
  }

  // Public methods dispatch events
  updateField = (field: string, value: string) => {
    this.add(new UpdateFieldEvent(field, value));
  };

  submit = () => {
    this.add(new SubmitFormEvent());
  };

  reset = () => {
    this.add(new ResetFormEvent());
  };
}
```

### Using in React

```tsx
function LoginForm() {
  const [state, bloc] = useBloc(FormValidationBloc);

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    if (state.isValid && !state.isSubmitting) {
      bloc.submit();
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={state.emailValue}
        onChange={(e) => bloc.updateField('email', e.target.value)}
        placeholder="Email"
      />
      {state.emailError && state.emailTouched && (
        <span>{state.emailError}</span>
      )}

      <input
        type="password"
        value={state.passwordValue}
        onChange={(e) => bloc.updateField('password', e.target.value)}
        placeholder="Password"
      />
      {state.passwordError && state.passwordTouched && (
        <span>{state.passwordError}</span>
      )}

      <button disabled={!state.isValid || state.isSubmitting}>
        {state.isSubmitting ? 'Submitting...' : 'Submit'}
      </button>
    </form>
  );
}
```

Notice how the flat state structure (`emailValue`, `passwordValue`, etc.) makes it easy to:
1. Access specific fields directly
2. Track which fields are touched
3. Update individual field errors
4. Check overall form validity

## Async Event Processing

Blocs handle async operations elegantly. You can **emit multiple states** during an async operation:

```typescript
interface Post {
  id: number;
  title: string;
  body: string;
}

interface ApiState {
  data: Post[] | null;
  loading: boolean;
  error: string | null;
  fetchCount: number;
}

class FetchDataEvent {}

class AsyncEventBloc extends Bloc<ApiState, FetchDataEvent> {
  constructor() {
    super({
      data: null,
      loading: false,
      error: null,
      fetchCount: 0,
    });

    this.on(FetchDataEvent, async (event, emit) => {
      // 1. Emit loading state immediately
      emit({
        ...this.state,
        loading: true,
        error: null,
      });

      try {
        // 2. Perform async work
        const response = await fetch('/api/posts');
        if (!response.ok) {
          throw new Error(`HTTP ${response.status}`);
        }
        const data = await response.json();

        // 3. Emit success state
        emit({
          data,
          loading: false,
          error: null,
          fetchCount: this.state.fetchCount + 1,
        });
      } catch (error) {
        // 4. Emit error state
        emit({
          ...this.state,
          loading: false,
          error: error instanceof Error ? error.message : 'An error occurred',
        });
      }
    });
  }

  fetchData = () => {
    this.add(new FetchDataEvent());
  };
}
```

The ability to emit multiple states during async work creates a smooth UX:
1. **Immediate feedback**: Loading state shows instantly
2. **Success handling**: Data appears when ready
3. **Error handling**: Errors displayed clearly
4. **No race conditions**: Each event runs to completion

<GuideArticleNote title="Cancellation Pattern">
For operations that can be cancelled (like search queries), store an `AbortController` and abort it when a new event arrives:

```typescript
class SearchBloc extends Bloc<SearchState, SearchEvent> {
  private abortController?: AbortController;

  constructor() {
    super(initialState);

    this.on(SearchEvent, async (event, emit) => {
      // Cancel previous request
      this.abortController?.abort();
      this.abortController = new AbortController();

      emit({ ...this.state, loading: true });

      try {
        const data = await fetch('/search', {
          signal: this.abortController.signal,
        });
        emit({ data, loading: false });
      } catch (error) {
        if (error.name !== 'AbortError') {
          emit({ ...this.state, loading: false, error: error.message });
        }
      }
    });
  }
}
```
</GuideArticleNote>

## Event Transformation: Debouncing

Sometimes you want to transform events before processing them. A common pattern is **debouncing** search queries:

```typescript
class SearchEvent {
  constructor(public readonly query: string) {}
}

interface SearchState {
  query: string;
  results: SearchResult[];
  isSearching: boolean;
  searchCount: number;
}

class SearchBloc extends Bloc<SearchState, SearchEvent> {
  private searchTimeout?: NodeJS.Timeout;

  constructor() {
    super({
      query: '',
      results: [],
      isSearching: false,
      searchCount: 0,
    });

    this.on(SearchEvent, async (event, emit) => {
      // Clear previous timeout
      if (this.searchTimeout) {
        clearTimeout(this.searchTimeout);
      }

      // Update query immediately for UI responsiveness
      emit({
        ...this.state,
        query: event.query,
        isSearching: event.query.length > 0,
      });

      if (event.query.length === 0) {
        emit({
          ...this.state,
          query: '',
          results: [],
          isSearching: false,
        });
        return;
      }

      // Debounce the actual search (500ms delay)
      await new Promise<void>((resolve) => {
        this.searchTimeout = setTimeout(() => resolve(), 500);
      });

      // Perform search after debounce
      const results = await this.performSearch(event.query);

      emit({
        query: event.query,
        results,
        isSearching: false,
        searchCount: this.state.searchCount + 1,
      });
    });
  }

  search = (query: string) => {
    this.add(new SearchEvent(query));
  };

  // Clean up on dispose
  override dispose() {
    if (this.searchTimeout) {
      clearTimeout(this.searchTimeout);
    }
    return super.dispose();
  }
}
```

This pattern:
- Updates UI immediately (responsive)
- Waits 500ms before searching (debounce)
- Cancels previous searches (efficiency)
- Cleans up timers (no memory leaks)

## Testing Blocs

Events make testing straightforward. You dispatch events and verify state changes:

```typescript
import { describe, it, expect, beforeEach } from 'vitest';

describe('EventCounterBloc', () => {
  let bloc: EventCounterBloc;

  beforeEach(() => {
    bloc = new EventCounterBloc();
  });

  it('should start at zero', () => {
    expect(bloc.state.count).toBe(0);
    expect(bloc.state.history).toEqual([]);
  });

  it('should increment by amount', () => {
    // Dispatch event
    bloc.add(new IncrementEvent(5));

    // Verify state
    expect(bloc.state.count).toBe(5);
    expect(bloc.state.history).toHaveLength(1);
    expect(bloc.state.history[0].event).toBe('Increment(5)');
  });

  it('should handle event sequence', () => {
    bloc.add(new IncrementEvent(3));
    bloc.add(new DecrementEvent(1));
    bloc.add(new IncrementEvent(2));

    expect(bloc.state.count).toBe(4);
    expect(bloc.state.history).toHaveLength(3);
  });

  it('should reset to zero', () => {
    bloc.add(new IncrementEvent(10));
    bloc.add(new ResetEvent());

    expect(bloc.state.count).toBe(0);
    expect(bloc.state.history).toHaveLength(2);
  });
});

describe('AsyncEventBloc', () => {
  it('should handle async events', async () => {
    const bloc = new AsyncEventBloc();

    // Start fetch
    bloc.add(new FetchDataEvent());
    expect(bloc.state.loading).toBe(true);

    // Wait for completion
    await waitFor(() => !bloc.state.loading);

    // Verify result
    expect(bloc.state.data).toBeDefined();
    expect(bloc.state.error).toBeNull();
    expect(bloc.state.fetchCount).toBe(1);
  });
});
```

Testing benefits:
- **Clear**: Dispatch event, verify state
- **Isolated**: Test each event handler independently
- **Reproducible**: Events create deterministic state transitions
- **Fast**: No React rendering needed

## Instance Lifecycle

Just like Cubits, Blocs follow automatic lifecycle management:

<GuideArticleNote title="Automatic Disposal">
**By default, BlaC counts component listeners**. Bloc instances automatically dispose when their listener count reaches zero:

- **Shared Blocs (default)**: Dispose when all components unmount
- **Isolated Blocs** (`static isolated = true`): Coupled to component lifecycle
- **Keep-alive Blocs** (`static keepAlive = true`): Must be manually disposed

You rarely need to manually dispose Blocs unless using `keepAlive = true`. Just unmount components, and BlaC handles cleanup.
</GuideArticleNote>

### Manual Cleanup for Resources

If your Bloc holds resources (timers, connections, subscriptions), override `dispose()`:

```typescript
class TimerBloc extends Bloc<TimerState, TimerEvents> {
  private intervalId?: NodeJS.Timeout;

  constructor() {
    super(initialState);

    this.on(StartTimerEvent, (event, emit) => {
      this.intervalId = setInterval(() => {
        emit({ ...this.state, seconds: this.state.seconds + 1 });
      }, 1000);
    });

    this.on(StopTimerEvent, (event, emit) => {
      if (this.intervalId) {
        clearInterval(this.intervalId);
        this.intervalId = undefined;
      }
    });
  }

  // Clean up resources
  override dispose() {
    if (this.intervalId) {
      clearInterval(this.intervalId);
    }
    return super.dispose();
  }
}
```

## When to Use Blocs vs Cubits

| Use Blocs When: | Use Cubits When: |
|----------------|------------------|
| Complex business logic with multiple triggers | Simple, direct state updates |
| You need event traceability | UI state (toggles, forms) |
| Testing is a high priority | Rapid prototyping |
| Multiple sources trigger same state change | Performance is critical |
| You want time-travel debugging | State changes are straightforward |
| Audit trail required | You prefer simplicity |

<GuideArticleNote title="Start Simple, Add Complexity When Needed">
**Recommendation**: Start with Cubits for most features. Upgrade to Blocs when you need:
- Event logging/debugging
- Complex async flows
- Multiple event sources
- Strict testability

You can always refactor a Cubit to a Bloc later. Don't over-engineer early!
</GuideArticleNote>

## Key Takeaways

- **Events are classes** that represent actions and carry data
- **Event handlers** registered with `on()` process events and emit state
- **Flat state** enables optimal dependency tracking at root level
- **Async events** can emit multiple states during execution
- **Event transformation** (debounce, throttle, cancel) is built into event handlers
- **Testing is straightforward** - dispatch events, verify state changes
- **Automatic disposal** when listener count reaches zero (unless keepAlive)
- **Choose Blocs** for complex logic, **Cubits** for simple state
- **Traceability** - every state change has a clear event cause
