# Bloc Lifecycle Management

Every Bloc and Cubit in your application has a **lifecycle**—from creation to disposal. Understanding this lifecycle is essential for building efficient, memory-safe applications. Let's explore how BlaC manages instances, when they're created and destroyed, and how to control their behavior.

## What You'll Learn

- The lifecycle stages: Creation → Active → Disposal
- Instance management patterns: Shared, Isolated, and Keep Alive
- Automatic disposal and memory management
- Cleanup with `onDispose` hooks
- Lifecycle monitoring with plugins

## Lifecycle Stages

A Bloc or Cubit goes through three main stages:

```typescript
// 1. CREATION
const cubit = new MyCubit(); // Constructor runs

// 2. ACTIVE
cubit.increment(); // State changes happen
const state = cubit.state; // State is accessed

// 3. DISPOSAL
cubit.dispose(); // onDispose hook runs, instance cleaned up
```

**Important:** In BlaC React apps, you rarely call `dispose()` manually. The framework handles disposal automatically based on component lifecycle and instance patterns.

## Automatic Disposal: When Instances Are Disposed

<GuideArticleNote title="Critical: Listener Count Drives Disposal">
BlaC tracks how many React components are **actively listening** to each Bloc/Cubit instance. When this listener count reaches **zero**, the instance is automatically disposed (unless `keepAlive = true`).

This is **not** the same as counting `useBloc` calls—only components that are **actively subscribed** are counted.
</GuideArticleNote>

### Shared Instances (Default Behavior)

By default, all components share a single instance:

```typescript
class UserCubit extends Cubit<User> {
  constructor() {
    super({ id: '', name: '' });
    console.log('[UserCubit] Created');
  }

  onDispose = () => {
    console.log('[UserCubit] Disposed');
  };
}

// Component A mounts
function ComponentA() {
  const [user, cubit] = useBloc(UserCubit);
  // First component creates the instance
  // Listener count: 1
  return <div>{user.name}</div>;
}

// Component B mounts
function ComponentB() {
  const [user, cubit] = useBloc(UserCubit);
  // Reuses the same instance
  // Listener count: 2
  return <div>{user.id}</div>;
}

// When Component A unmounts → Listener count: 1 (instance still alive)
// When Component B unmounts → Listener count: 0 → DISPOSAL HAPPENS
```

**Key insight:** The shared instance survives as long as **at least one component** is subscribed. When **all** components unmount, the listener count reaches zero and disposal occurs.

### Initialization: When Is an Instance Created?

An instance is created **when the first component calls `useBloc`**:

```typescript
class ShoppingCartCubit extends Cubit<CartState> {
  constructor() {
    super({ items: [], total: 0 });
    console.log('Cart created!');
  }
}

// Before any component mounts: instance doesn't exist
// First component calls useBloc → constructor runs
// Subsequent components → reuse the same instance
```

<GuideArticleNote title="Lazy Initialization">
Bloc/Cubit instances are created **lazily**—only when needed. They don't exist until a component requests them.
</GuideArticleNote>

## Instance Patterns

BlaC supports three instance management patterns, each suited to different use cases.

### Pattern 1: Shared (Default)

**All components share one instance.**

```typescript
class ThemeCubit extends Cubit<'light' | 'dark'> {
  // No special configuration needed
  constructor() {
    super('light');
  }
}
```

**When to use:**
- App-wide state (theme, user auth, settings)
- Global services (API client, notifications)
- State that should be synchronized across components

### Pattern 2: Isolated

**Each component gets its own private instance.**

```typescript
class FormCubit extends Cubit<FormState> {
  static isolated = true; // Each component gets its own instance

  constructor() {
    super({ email: '', password: '' });
  }
}

// Component A
function LoginForm() {
  const [form, cubit] = useBloc(FormCubit);
  // Gets instance #1
}

// Component B
function SignupForm() {
  const [form, cubit] = useBloc(FormCubit);
  // Gets instance #2 (completely independent)
}
```

**When to use:**
- Form state (each form needs independent state)
- Modal/dialog state
- Component-specific UI state
- Any state that should NOT be shared

**Disposal:** Each isolated instance is disposed when **its specific component** unmounts (listener count reaches zero for that instance).

### Pattern 3: Keep Alive

**Instance persists even when all components unmount.**

```typescript
class CacheCubit extends Cubit<CacheState> {
  static keepAlive = true; // Survives unmounting!

  constructor() {
    super({ data: {}, lastFetch: null });
  }

  onDispose = () => {
    // Only called on manual disposal
    console.log('Manually disposed');
  };
}

// Component can mount/unmount freely
function DataView() {
  const [cache, cubit] = useBloc(CacheCubit);
  // State persists between mounts!
}

// Manual disposal (when really needed):
// Blac.dispose(CacheCubit);
```

**When to use:**
- Caching layer that should persist
- Background tasks/timers
- Session state
- User preferences

**Warning:** Keep-alive instances stay in memory until manually disposed. Use sparingly!

<GuideArticleNote title="Memory Management with Keep Alive">
Keep-alive instances **never automatically dispose**. They accumulate in memory until you manually call `Blac.dispose(MyCubit)`.

Only use `keepAlive = true` when you truly need persistent state. For most use cases, the default shared pattern is better.
</GuideArticleNote>

## The `onDispose` Hook

Use `onDispose` to clean up resources when an instance is disposed:

```typescript
class ResourceCubit extends Cubit<State> {
  private subscription?: Subscription;
  private timer?: NodeJS.Timeout;
  private controller?: AbortController;

  constructor() {
    super(initialState);

    // Set up resources
    this.subscription = eventBus.subscribe(this.handleEvent);
    this.timer = setInterval(this.tick, 1000);
    this.controller = new AbortController();

    this.fetchData();
  }

  onDispose = () => {
    console.log('Cleaning up resources...');

    // Unsubscribe from events
    this.subscription?.unsubscribe();
    this.subscription = undefined;

    // Clear timers
    if (this.timer) {
      clearInterval(this.timer);
      this.timer = undefined;
    }

    // Cancel pending requests
    this.controller?.abort();
    this.controller = undefined;

    // Any other cleanup
    console.log('All resources cleaned up');
  };

  private fetchData = async () => {
    try {
      const response = await fetch('/api/data', {
        signal: this.controller?.signal,
      });
      const data = await response.json();
      this.emit({ ...this.state, data });
    } catch (error) {
      if (error.name !== 'AbortError') {
        console.error('Fetch error:', error);
      }
    }
  };

  private tick = () => {
    this.patch({ timestamp: Date.now() });
  };

  private handleEvent = (event: Event) => {
    this.patch({ lastEvent: event });
  };
}
```

<GuideArticleNote title="Always Clean Up External Resources">
If your Bloc/Cubit uses:
- **Timers** → `clearInterval()` / `clearTimeout()`
- **Subscriptions** → `unsubscribe()`
- **Network requests** → `AbortController.abort()`
- **WebSockets** → `socket.close()`
- **Event listeners** → `removeEventListener()`

Then you **must** clean them up in `onDispose` to prevent memory leaks.
</GuideArticleNote>

## Disposal Checklist

When writing a Cubit or Bloc, ask yourself:

1. **Do I create timers/intervals?** → Clear them in `onDispose`
2. **Do I subscribe to external events?** → Unsubscribe in `onDispose`
3. **Do I create network connections?** → Close them in `onDispose`
4. **Do I use AbortController?** → Abort in `onDispose`
5. **Do I store large data?** → Clear references in `onDispose`

If you answered "yes" to any of these, implement `onDispose`:

```typescript
class WellBehavedCubit extends Cubit<State> {
  private resources: Resource[] = [];

  constructor() {
    super(initialState);
    // Set up resources
  }

  onDispose = () => {
    // Clean up ALL resources
    this.resources.forEach(r => r.cleanup());
    this.resources = [];
  };
}
```

## Lifecycle Monitoring with Plugins

BlaC plugins can monitor all lifecycle events across your application:

```typescript
import { BlacPlugin, Blac } from '@blac/core';

class LifecycleLoggerPlugin implements BlacPlugin {
  name = 'LifecycleLogger';
  version = '1.0.0';

  onBlocCreated(bloc: any): void {
    console.log(`✅ CREATED: ${bloc.constructor.name}`);
  }

  onBlocDisposed(bloc: any): void {
    console.log(`🗑️ DISPOSED: ${bloc.constructor.name}`);
  }

  onStateChanged(bloc: any, prevState: any, newState: any): void {
    console.log(`🔄 STATE CHANGED: ${bloc.constructor.name}`);
  }
}

// Register globally
Blac.instance.plugins.add(new LifecycleLoggerPlugin());
```

**Use cases for lifecycle plugins:**
- Development debugging
- Performance monitoring
- Analytics tracking
- DevTools integration
- Memory leak detection

## Common Patterns

### Pattern: Cleanup Timers

```typescript
class TimerCubit extends Cubit<number> {
  private interval?: NodeJS.Timeout;

  constructor() {
    super(0);
  }

  startTimer = () => {
    // Always clear existing timer first
    this.stopTimer();

    this.interval = setInterval(() => {
      this.emit(this.state + 1);
    }, 1000);
  };

  stopTimer = () => {
    if (this.interval) {
      clearInterval(this.interval);
      this.interval = undefined;
    }
  };

  onDispose = () => {
    this.stopTimer(); // Essential cleanup!
  };
}
```

### Pattern: Abort Async Operations

```typescript
class DataFetchCubit extends Cubit<DataState> {
  private controller = new AbortController();

  constructor() {
    super({ data: null, loading: false, error: null });
  }

  fetchData = async () => {
    this.patch({ loading: true, error: null });

    try {
      const response = await fetch('/api/data', {
        signal: this.controller.signal,
      });

      const data = await response.json();
      this.patch({ data, loading: false });
    } catch (error) {
      // Don't set error state if aborted (component unmounted)
      if (error.name !== 'AbortError') {
        this.patch({ error: error.message, loading: false });
      }
    }
  };

  onDispose = () => {
    this.controller.abort(); // Cancel in-flight requests
  };
}
```

### Pattern: Subscription Management

```typescript
class EventStreamCubit extends Cubit<Event[]> {
  private subscription?: Subscription;

  constructor() {
    super([]);
    this.subscription = eventBus.subscribe((event) => {
      this.emit([...this.state, event]);
    });
  }

  onDispose = () => {
    this.subscription?.unsubscribe();
    this.subscription = undefined;
  };
}
```

## React Integration: Component Lifecycle

How Bloc lifecycle integrates with React:

```typescript
function MyComponent() {
  // Component mounts → Listener count increases
  const [state, cubit] = useBloc(MyCubit);

  useEffect(() => {
    // Component effect runs
    cubit.initialize();

    return () => {
      // Component cleanup runs
      cubit.cleanup();
    };
  }, [cubit]);

  // Component unmounts → Listener count decreases
  // If count reaches 0 → cubit.onDispose() called
}
```

<GuideArticleNote title="Component Cleanup vs Bloc Disposal">
These are **different lifecycle hooks**:

- **React `useEffect` cleanup**: Runs when the component unmounts
- **Bloc `onDispose`**: Runs when the **instance** is disposed (when listener count reaches zero)

For shared instances, the React cleanup may run many times (as different components unmount), but `onDispose` only runs once when the **last listener** unmounts.
</GuideArticleNote>

## Testing Lifecycle

Test lifecycle behaviors explicitly:

```typescript
import { describe, test, expect, vi } from 'vitest';

describe('Cubit lifecycle', () => {
  test('constructor initializes state', () => {
    const cubit = new MyCubit();
    expect(cubit.state).toEqual(expectedInitialState);
    cubit.dispose();
  });

  test('onDispose cleans up resources', () => {
    const cubit = new TimerCubit();
    cubit.startTimer();

    // Spy on cleanup
    const clearIntervalSpy = vi.spyOn(global, 'clearInterval');

    cubit.dispose();

    expect(clearIntervalSpy).toHaveBeenCalled();
    cubit.dispose(); // Cleanup
  });

  test('disposal happens after all consumers unmount', () => {
    const cubit = new SharedCubit();
    const disposeSpy = vi.spyOn(cubit, 'onDispose');

    // Simulate multiple consumers
    cubit._incrementConsumerCount(); // Consumer 1
    cubit._incrementConsumerCount(); // Consumer 2

    cubit._decrementConsumerCount(); // Consumer 1 unmounts
    expect(disposeSpy).not.toHaveBeenCalled(); // Still alive

    cubit._decrementConsumerCount(); // Consumer 2 unmounts
    expect(disposeSpy).toHaveBeenCalled(); // Now disposed
  });
});
```

## Common Pitfalls

### Pitfall 1: Forgetting to Clean Up

```typescript
// ❌ BAD: Memory leak!
class BadCubit extends Cubit<State> {
  constructor() {
    super(initialState);
    setInterval(() => {
      this.emit(newState); // Runs forever!
    }, 1000);
  }
  // No onDispose → timer never cleared
}

// ✅ GOOD: Proper cleanup
class GoodCubit extends Cubit<State> {
  private timer?: NodeJS.Timeout;

  constructor() {
    super(initialState);
    this.timer = setInterval(() => {
      this.emit(newState);
    }, 1000);
  }

  onDispose = () => {
    if (this.timer) {
      clearInterval(this.timer);
      this.timer = undefined;
    }
  };
}
```

### Pitfall 2: Relying on Disposed Instances

```typescript
// ❌ BAD: Using disposed instance
function MyComponent() {
  const [state, cubit] = useBloc(MyCubit);

  const handleClick = () => {
    setTimeout(() => {
      cubit.doSomething(); // Component may have unmounted!
    }, 5000);
  };
}

// ✅ GOOD: Check if still mounted
function MyComponent() {
  const [state, cubit] = useBloc(MyCubit);
  const mountedRef = useRef(true);

  useEffect(() => {
    return () => {
      mountedRef.current = false;
    };
  }, []);

  const handleClick = () => {
    setTimeout(() => {
      if (mountedRef.current) {
        cubit.doSomething();
      }
    }, 5000);
  };
}
```

### Pitfall 3: State Updates During Disposal

```typescript
// ❌ BAD: Emitting state during disposal
class BadCubit extends Cubit<State> {
  onDispose = () => {
    this.emit(finalState); // No listeners to receive it!
    this.cleanup();
  };
}

// ✅ GOOD: Just clean up
class GoodCubit extends Cubit<State> {
  onDispose = () => {
    // No state emissions—just cleanup
    this.cleanup();
  };
}
```

### Pitfall 4: Over-Using Keep Alive

```typescript
// ❌ BAD: Everything is keep-alive
class EverythingCubit extends Cubit<State> {
  static keepAlive = true; // Do I really need this?
}

// ✅ GOOD: Only persistent state
class CacheCubit extends Cubit<CacheState> {
  static keepAlive = true; // Yes, cache should persist
}

class FormCubit extends Cubit<FormState> {
  // No keepAlive - form state can dispose
}
```

## Decision Matrix: Choosing Instance Patterns

```
Do all components need to share the same state?
│
├─ YES → Use Shared (default)
│         ✅ User auth
│         ✅ App theme
│         ✅ Global settings
│
└─ NO → Should state persist when all components unmount?
        │
        ├─ YES → Use Keep Alive
        │         ✅ Cache
        │         ✅ Background tasks
        │         ✅ Session data
        │
        └─ NO → Use Isolated
                  ✅ Forms
                  ✅ Modals
                  ✅ Component-specific state
```

## Key Takeaways

- **Automatic disposal** happens when listener count reaches zero
- **Shared instances** (default) survive as long as any component is subscribed
- **Isolated instances** give each component its own private state
- **Keep-alive instances** persist until manually disposed
- **Always implement `onDispose`** when using external resources
- **Lifecycle plugins** enable monitoring and debugging
- **Cleanup is critical** for preventing memory leaks
