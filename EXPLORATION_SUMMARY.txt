ISSUE #1: CIRCULAR DEPENDENCIES (Partially Mitigated)
────────────────────────────────────────────────────────

Status: PARTIALLY FIXED (but 22 unsafe type assertions remain)

Problem:
  - BlocBase.ts imports Blac (for logging)
  - Blac.ts imports BlocBase (for instance management)
  - Creates circular import at module load

Current Mitigation:
  ✓ Introduced BlacContext interface (breaks circular dependency)
  ✓ BlocBase depends on interface, not Blac class
  ✓ Blac injects itself after instantiation

Remaining Issues:
  ✗ 22+ unsafe `as any` type assertions still in code
  ✗ Example: (bloc as any)._disposalState
  ✗ Example: (this._plugins as any).transformEvent(action)

Affected Files:

Recommendation: Extract private methods to public accessors

ISSUE #5: OVER-ENGINEERING SIGNS
─────────────────────────────────

A. PathTrie (152 LOC)
   Purpose: Filter leaf paths from set
   Implementation: Trie data structure
   Optimization: O(n) vs O(n²) nested loops
   
   Concerns:
   - Only used in React integration (not core)
   - No benchmarks showing necessity
   - Would simple array filter work?
   - Typical set sizes unknown
   
   Assessment: Possibly premature optimization

B. PathIndex (270 LOC)
   Purpose: O(1) path relationship queries
   Implementation: Pre-computed path nodes
   
   Concerns:
   - Complex structure (parent-child, ancestors, depth)
   - Significant memory overhead
   - Real benefit unclear
   - Maintenance burden
   
   Assessment: May not justify complexity

C. Generation Counter Pattern
   Location: /src/lifecycle/BlocLifecycle.ts
   Purpose: Prevent disposal race conditions
   
   Implementation:
   - Increment generation on disposal request
   - Microtask captures generation
   - Validates before executing
   
   Concerns:
   - Adds complexity to fragile system
   - Race condition tests needed (detailed)
   - Simpler alternatives might exist
   
   Assessment: Necessary but adds complexity
